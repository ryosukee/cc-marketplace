# 入り組んだ説明・報告・確認は HTML で行う

調査報告・設計判断の比較・設問の多い確認など、前提や構造が入り組んだ内容は、
ターミナルへのテキスト出力ではなく self-contained な HTML ページを生成して見せる。
構造化して順序立てて読める形に作り込むこと自体が目的なので、体裁は手を抜かない。

## 適用基準

- HTML にする: 前提説明が長い調査報告、複数の表・比較を含む設計判断、設問が概ね 4 問を超える確認、順序立てた説明が必要なもの
- テキストのまま: 短い報告、単発〜少数の質問（レンダリングバグ修正までは AskUserQuestion を使わず、
  チャットのフリーテキストで質問する。[ユーザーへの確認は選択肢形式で](./ask-with-choices.md) 冒頭の警告参照）

## 生成・表示方法

- scratchpad ディレクトリに kebab-case の `.html` を書き、`open -a "Google Chrome" <file>` で Chrome で開く
- 開き直せるように、ファイルパスを本文でも伝える
- CSS/JS はすべてインラインで self-contained にする。外部 CDN・フォント・画像に依存しない
- `prefers-color-scheme` でライト/ダーク両対応にする
- 構成は「結論・概要 → 前提・経緯 → 詳細 →（あれば）設問」の順。セクション番号か目次を付ける
- **大セクションの見出しには「2/5」のように全体数込みの番号を付ける**。読者が全体の分量と現在位置を把握できるようにする
- ターン構造と無関係にブラウザで表示されるため、
  [ユーザーへの確認は選択肢形式で](./ask-with-choices.md) に記載のレンダリングバグの影響を受けない

## 情報設計（構成・テーブル・羅列）

情報デザイン norm ライブラリ（product-boilerplate repo の `docs/norms/`、出典付き）に従う。
作成時に特に効く要点（詳細・出典はライブラリ側）:

- 冒頭に地図を置く: この文書の目的 / 決めること（設問）一覧 / 読み方。結論・答えは背景より前に出す
- 1 議題 1 セクション。判断セクションは「決めること → 背景 → 判断材料 → 設問」の同型に揃える
- すべての表・グリッド・カード群の直前にリード文を置く: 「これは何の一覧か」+「どこが決め手か」
- テーブルのセルは流し読みできる短い判定値（体言止め・記号・数値）にし、文を書かない。
  根拠・詳細は表直下の補足リストか脚注へ出す。1 セル 1 情報
- 比較表は「列 = 選択肢、行 = 観点」で列は 5 以下。決め手の行はハイライトし、リード文で行名を名指しする
- 出典・長い補足を本文の括弧書きで埋め込まない。出典はチップ等に分離する
- 回答・判断に不要な資料は末尾の参考資料セクションへ隔離し、「読み飛ばしてよい」と明記する
- 推奨バッジには根拠の併記と覆し可能性（非推奨側も選べることの明示）を必須にする

## 確認（設問）を含めるときの作り

- **設問は関連するセクションの直下に置く**。ページ末尾にまとめて並べない
    - 設問が多いとき、末尾集約は判断材料と設問の往復スクロールが発生して回答しづらい
    - 回答コピー機能は全セクションの設問を集約して 1 テキストにする（配置と集約は別）
- **最終確認などの自由記入欄は、ページ末尾ではなくスクロール追従の下部固定領域に置く**（コピー用フッター内など）。
  上から順に読みながら、気づいた点を都度記入できるようにする
- 各設問に選択肢 (radio / checkbox) を用意し、推奨案には推奨バッジを付ける
- **技術選定などの比較設問には、選択肢の説明文だけで判断させない**。設問の直前に多観点の比較表
  (機能・性能・実績・エコシステム・リスク・移行性・保守状況など、その選定で効く観点) を置き、
  各観点は調査の裏取り付きで書く。選択肢の description は比較表の結論の要約に徹する
- 選択肢に収まらない回答用に、設問ごとに「その他」の自由記述欄を付ける
- 設問ごとに補足記入欄（任意）を付ける
- ページ末尾に回答の live preview と「回答をコピー」ボタンを置く。
  ボタンはそれまでの選択・自由記述を 1 つのテキストに整形してクリップボードへコピーする
- 整形フォーマットは Claude Code にそのまま貼れる形にする:

  ```text
  ## HTML フォーム回答（{テーマ}）
  - Q1（{設問}）: {回答}
  - Q2（{設問}）: {回答}
  - 補足: {自由記述}
  ```

- クリップボード API が使えない環境向けに、preview の textarea を全選択コピーできるフォールバックを付ける
- 未回答の設問がある状態でもコピーは許可し、未回答は `未回答` と明記する

## フォームを op-review でレビューする際の facet 分担と蓄積 norm

フォーム類の op-review は 1 agent に全部見せず、facet を分担した並列レビューにし、
全 facet が approve（指摘 0 件）になるまで修正を繰り返してから提示する:

- 内容整合 facet: 事実・数値の突合、上流文書との整合、確定事項の峻別。norm = 下記の蓄積 norm + 上流文書
- 情報デザイン facet: 可読性・情報設計のみ。norm = 情報デザイン norm ライブラリ
  （product-boilerplate repo の `docs/norms/`。S / T / F / V 系 + 各ファイル末尾の機械検証手順）
- clarity facet: 一意に解釈できるか・判断材料が足りているか・未定義用語がないかのみ。
  norm = 同ライブラリの C 系（clarity-communication.md）。
  **経緯・設計文脈を知らない新規 agent で毎回実行する**（文脈なしで読んで伝わるかの外部検証。
  経緯を知る agent は書き手と同じ curse of knowledge を持ち、未定義用語・文脈依存の記述に気づけない）

分担の根拠: 単一 agent に全観点を見せると観点が薄まり、facet を分けると異なる種類の指摘が独立に出る。
また、経緯を知る agent は AI が導入した未定義用語や文脈依存の記述を素通りする
（例: 無定義の独自用語が複数版のレビューを通過し、ユーザーレビューで初めて発覚した事例）。
これが文脈なし agent による clarity facet 分離の根拠。

内容整合 facet には、普遍原則に加えて以下の蓄積観点を norm に含める
（ユーザーレビューで繰り返し指摘された観点。今後も指摘があればここに追記する）:

- 設問の自立性: 設問が参照する定義・分類・根拠はすべて同一文書内に再掲する（置き換え型フォームは特に）
- 論点の統合: 相互依存する論点を複数の設問に分けない。1 議題 1 設問
- 確定と未確定の峻別: 確定事項を無断で再質問・変更しない。覆す選択肢には明示フラグ。確定していないものを確定扱いしない
- 一括確認枠の掲載条件: 一括承認の枠に入れてよいのは確定事項からの変更を含まない項目（純粋なバージョン更新等）のみ。
  既存構成からの乗り換え・覆しは変更の大小を問わず個別設問 + 明示フラグで扱う
- 覆しフラグは選択肢単位で付与する: 1 設問に覆しに該当する選択肢が複数ある場合、その全選択肢の説明文にフラグを付ける。
  設問文での言及はフラグの代替にならない
- 管理物の増加への警戒: 専用ファイル・独立 skill 等の新規管理物を増やす提案は、増減の検算と負債リスクの明示を伴う
- 最小ロードの粒度: rule 等の適用範囲は「本当にそれを必要とする対象」に絞られているかを疑う（広域ひとまとめを許さない）
- 実例は実測値そのまま: 実在 repo に帰属させる例示は実際のパス・値を転記し、理想化するなら架空例と明示
- 事実主張の正確さ: 数値・列挙は実測と突合。個数表記は同一文書内の列挙個数とも突合する。省略には省略マーカー
- 対話残留の排除: 「あなた」等の対話依存表現を使わない。ユーザー発言は「第N回指示の『…』」形式で引用
- 引用の verbatim 徹底: 「」で括る引用は回答原文からそのままコピーする（部分文字列として原文に存在するかで検算）。
  中間を省く場合は「…（中略）…」を明示する。圧縮したい場合は「」を使わず地の文で要約する
- 処遇の単一正: 対象ごとの処遇（移行対象・スコープ内外等）は設問セクションを唯一の正とし、
  他セクションでは結論を再記述せず「→ N.n 参照」のポインタにする。再記述する場合は文言一致を検算する
- 論点移動時の波及検索: 設問をセクション間で移動・分離したら、本文だけでなく placeholder・例文・注記も含めて
  移動した論点名で artifact 全体を検索し、旧配置前提の参照を更新する
- 確定チェーンの突合: 「確定事項の記録」は直前回で確定した項目を先に列挙し 1:1 で突合してから書く。
  部分確定（「ここまではよい」等）は対象範囲を項目単位に分解して漏れなく記録する
- 変更点の明示: 過去回の内容から変えた箇所を「〜のまま」と書かない。変更として明示し確認範囲に含める。
  無変更の項目も「（確定済み: {内容の要約}、変更なし）」の形で内容を再掲する
- 網羅性: 遷移表・仕分け表は対象全件を列挙してから分類する
- TOC の機械導出: 目次のリンクテキストはセクション見出しの文字列をそのまま複製する（別文言で要約しない）
- メタデータの突合: 回数・問数等のカウント改訂時は本文だけでなく title 等のメタデータ領域も grep で全出現箇所を突合する
- 量的表現のマーカー: 具体数値（レイテンシ・サイズ・期間等）は「裏取り済み」か「推定（未実測）」かを明示する
- 用語の出自: AI が持ち込んだ命名・専門用語はユーザー定義と区別し、初出で必ず定義する。
  過去回の選択肢文や図表の注記を経由して定着した用語を「定義済み」扱いしない
  （レビュー agent は経緯を知っているため気づけない。用語ごとに「ユーザーの発言に遡れるか」で検算する）
- 新語導入の規律: 概念に名前を付ける前に、既存の一般用語・業界標準語を調査し、あればそれを使う。
  独自のコイン語・分類コードを作らない。導入する場合はユーザーの明示合意を取る
  （clarity norm C12〜C13。例: 独自語「スロット化」→ 一般語「パラメータ化」に置き換えた類の事例）

## 回答の受け取り

ユーザーから「## HTML フォーム回答」で始まるテキストが貼られたら HTML フォームの回答として扱う。
以降の解釈まとめ・作業フローは [ユーザーへの確認は選択肢形式で](./ask-with-choices.md) の
「複数回の質問と回答後のフロー」と同じ。
