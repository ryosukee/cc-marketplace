# Markdown 記法・書式ガイド

`.md` ファイルを生成・編集する際の記法・表記ルール。
この rule が扱うのは Markdown 固有の記法だけで、日本語の文章術（構成・冗長排除・文体・AI 口調の禁止）は
常時 rule「日本語テキストの執筆要点」と japanese-text-writing skill に従う。

## 強調・装飾

- 太字（`**...**`）は基本的に使わない。例外は用語の初出定義（定義される語を太字にする）と、
  論理の要所（誤読を防ぐ否定など。一節に一、二箇所まで）
- 太字見出し（`**...**:`）で段落を区切らない。構造を出したいならサブセクション (`###`) か箇条書きのネストにする
- `—`（em dash, U+2014）はどの用途でも使わない。説明の接続は箇条書きのネストや補足ブロックで書く
- `§`（セクション記号）は使わない。セクションへの参照はアンカーリンク `[ドキュメント名](path.md#anchor)` で書く
- 絵文字を装飾目的で使わない。ステータス表現も絵文字ではなくテキストで書く

## 改行・余白

- 明示的な改行には `<br>` タグを使う。末尾スペース 2 つによる改行は使わない
    - 末尾スペースは不可視で、エディタの trailing whitespace 除去や Claude の Edit ツールで消されやすい
- 情報密度の高いブロックの直後は `<br>` や空行で視覚的な余白を意識する

## 行の折り返し位置

1 行が長くて折り返しが必要な場合、句点（。）または読点（、）の直後で改行する。
フレーズの途中（括弧内、修飾語句の直後、助詞の後など）で機械的に分割しない。

長文 1 行に句読点が足りない場合は、文を 2 文に分割してから改行する。
句読点以外で切るのは、コードブロック内のコメント・URL・固有名詞列など分割不可な要素がある場合のみ。

## 和欧混植

日本語サイエンスライティングの和欧混植ルールに準拠する。ここに明示されていないケースも慣例に従う。

### スペーシング

日本語と半角英数が隣接する場合は半角英数の前後に半角スペースを入れる。

- OK: `task skill と reference skill の 2 種類がある`
- NG: `task skillとreference skillの2種類がある`

例外（スペースを入れない）:

- 句読点の直前: `これがその skill。`
- インラインコード前後: `` `claude -p` で実行する``（バッククォート自体が視覚的区切りになる）

### 括弧

- 括弧内が半角英数 → 半角括弧。外側にスペース、内側にスペースなし: `セッション (Cookie) とは無関係`
- 括弧内が日本語等の全角文字 → 全角括弧: `Auth0 セッション（最大 7 日）より`

## インデント

- 箇条書きのネストは 4 スペースインデント。2 スペースだと renderer によってはネストと解釈されない

## 引用ブロック・参照

引用ブロック (`>`) の用途を以下に限定する。それ以外の目的で引用ブロックを使わない。

### 補足ブロック

本文の理解を助ける補足説明に使う。

- 1 行目: `> 補足: タイトル<br>`（`<br>` タグで改行）
- 2 行目: `> 本文...`
- タイトルに `**`（太字）は使わない
- 1 つの補足ブロックは 2 行（タイトル + 本文）で構成し、複数のトピックを説明しない

```markdown
> 補足: Confidential Client<br>
> OAuth 2.0 では...（本文）
```

### 引用

外部ドキュメントの文言をそのまま引用する。補足説明は加えず、文言の転記と出典のみ。

- 引用文は `"..."` で囲む
- 出典は `出典: [タイトル](URL)` の形式で最終行に置く

```markdown
> "Each execution of a trigger must complete in 20 seconds or less."
>
> 出典: [Actions Limitations](https://auth0.com/docs/customize/actions/limitations)
```

### 情報源ブロック

本文の主張や判断を支えるエビデンスを記録する。補足説明を加えながら情報源を示す。

- 1 行目: `> 情報源: タイトル<br>`
- 2 行目以降: 補足説明、根拠、リンク等
- リンク先のどこを根拠にしているかを本文中に書く

### GFM Alerts

注記・注意書き・アーカイブ注記など、読者への通知に使う。

- 使えるタイプ: `NOTE`, `TIP`, `IMPORTANT`, `WARNING`, `CAUTION`
- 主な用途: ドキュメントのステータス、セクションのスコープ注記、注意事項

### ※ 参照

関連ドキュメントへの参照先を示す。インラインリンクで地の文に組み込むのが基本だが、
うまく書けない場合に本文ブロックの直後に置く。

- `※ 説明文（[参照先タイトル](URL)）` の形式
- 複数ある場合は `※` を連続して書き、各行の末尾に `<br>` タグで改行する

### 配置と使い分け

引用ブロック（補足・情報源・引用）は、それが支える対象の記述の後に置く。

| 本文との関係 | 記法 |
| --- | --- |
| 理解を助ける説明 | `> 補足:` |
| 主張を支えるエビデンス | `> 情報源:` |
| 外部文言のそのままの転記 | `> "..."` + `> 出典:` |
| 読者への通知 | `> [!NOTE]` 等 |
| 参照先の提示 | `※` |

## ファイル間リンク

関連する記述や参照が必要になったら、必ず inline link または引用ブロック系でリンクする。
プレーンテキストでの `.md` ファイル名の言及は禁止。

- リンク文字列にはファイル名ではなく、対象ドキュメントの `# タイトル`（h1）をそのまま使う
- パスは相対パスで、同一ディレクトリなら `./`、別ディレクトリなら `../dir/` を使う。`./` プレフィックスは必須
- セクションへの参照はアンカーリンクにする。`[Doc](path.md#3-セクション名)` と書く
- バッククォート（`` `file.md` ``）での言及はリンクにならないため使わない
- タイトルを変更した場合は、被リンク側のリンク文字列もすべて更新する
- 「関連ドキュメント」テーブルは本文中で参照したドキュメントの集約インデックスとして使う。
  テーブルにしか載っていない参照は禁止

## lint エラーの修正方針

markdownlint のエラーには形式的に直せるものと、意味的な解決が必要なものがある。

形式的修正で済むもの（機械的に置換してよい）:

- em dash → 括弧や箇条書きに置き換え
- `./` プレフィックスの追加

意味的な解決が必要なもの（表面的な書き換えで lint を黙らせてはいけない）:

- TODO 残存: TODO の内容を実際に解決するか、未解決事項として管理する
- プレーンテキスト .md 参照: 適切なリンクテキストと相対パスを確認してリンク化する
- 壊れたアンカー: 対象ファイルの見出しを確認して正しいアンカーに修正する
