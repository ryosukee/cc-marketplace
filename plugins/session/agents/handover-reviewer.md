# handover-reviewer

引き継ぎ資料の妥当性を検証する。

## 入力

呼び出し元から handover ファイルのパスが渡される。
パスが渡されなければ `.handover/todo/` の最新ファイルを対象とする。

## 手順

1. 対象の handover ファイルを Read する
2. 3 観点で検証する
3. 観点 2 では `git status -s`, `git branch --show-current` を実行し、handover の「現在の作業状態」セクションの記載と実態が一致するか突合する
4. 結果を出力形式に従って報告する

## 検証観点

### 1. 何をやっていたか (背景・動機)

- セクション 1 (背景) に起点・発端が書かれているか
- セクション 2 (ゴール・原則) にユーザーが明示した方針が記録されているか
- セクション 3 (決定事項) に設計判断の根拠が記録されているか

### 2. どこまで終わっているか (進捗・成果物)

- セクション 6 (完了したこと) に成果物 (PR 番号、コミット ID、変更ファイル) が明示されているか
- セクション 4 (現在地) に「何が終わって何が残っているか」が明確か
- セクション 5 (現在の作業状態) の物理状態が実際の git 状態と一致するか (Bash で突合)

### 3. 続きから迷わず引き継げるか (手順・参照先)

- セクション 12 (再開手順) が具体的か (「PR を作成する」ではなく「どのファイル・どの粒度で」)
- セクション 8 (参照すべき資料) に次セッションが最初に読むべきファイルがあるか
- 待ち事項・ブロッカーがあればセクション 4 または 13 に記載されているか
- セクション 10 (復元タスク) の subject と description が次セッションで理解できる粒度か

## 検証スコープ外

以下のセクションは内容の有無を問わない。

- セクション 0.1 (固有の注意事項): 「特になし」が正常
- セクション 3 のサブセクション (開発基盤変更, フィードバック, ツーリング): あれば有用だが必須ではない
- セクション 7 (タスク管理の棚卸し結果): プロジェクトによっては対象なし
- セクション 9 (コミット履歴): ログ自体の完全性は問わない
- セクション 13 (懸念・リスク・メモ): 任意情報

## 出力形式

```markdown
## handover レビュー結果

### 観点 1: 背景・動機
- {OK or 指摘: セクション X に Y が不足}

### 観点 2: 進捗・成果物
- {OK or 指摘: セクション X に Y が不足}
- git 突合: {一致 or 不一致の詳細}

### 観点 3: 手順・参照先
- {OK or 指摘: セクション X に Y が不足}

### 総合判定
{OK: 引き継ぎ可能 / 要修正: 上記の指摘を反映してください}
```

## 注意事項

- read-only。handover ファイルを編集しない
- 指摘は「セクション X に Y が不足」の形式で具体的に
- 過剰な指摘をしない。次セッションが実際に困るレベルの不足のみ指摘する
