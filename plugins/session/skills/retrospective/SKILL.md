# retrospective

セッションで得た学びを rules/skills/agents/CLAUDE.md に codify する。
学びの抽出・明文化のみを担う。

## codify (rules/skills/CLAUDE.md の更新 → コミット)

### ステップ 1: 棚卸し

以下のソースから codify 候補を洗い出す。

#### セッション内のやり取り

時系列ではなくテーマごとに整理する。

- 何が繰り返し起きたか
- 何が実害になったか (バグ、差し戻し、やり直し、事故)
- どんな判断ルールが発生したか
- どんなワークフローが機能した / しなかったか

#### 既存ドキュメントの陳腐化

`.claude/rules/`, `.claude/skills/`, `.claude/agents/`,
`CLAUDE.md`, `README.md` をざっと確認し:

- セッション中に使った rules/skills が実態と乖離していないか
- ユーザーの指摘・訂正のうちルール化されていないものはないか
- 使われていないセクション・機能はないか

#### skill/rule の妥当性検証

このセッションで実際に発動・参照した skill/rule を対象に検証する。

- skill の手順: 記述通りに動いたか。実運用と乖離したステップはないか
- skill の trigger (description): 意図した場面で発動したか
- rule の制約: 実際の開発で遵守できたか
- agent の定義: 期待通りの振る舞いをしたか

#### 今回の作業の反映漏れ

セッション中に行った変更が、反映すべき関連箇所に行き渡っているかを確認する。
反映先はプロジェクトによって異なる。

#### auto memory の棚卸し

memory に蓄積された知見のうち、rule や skill に昇格すべきものがないかチェックする。

### ステップ 2: 優先度分類

| 優先度 | 基準 | 対応 |
| --- | --- | --- |
| A | 実害があった + 将来再現する | 強く推奨 |
| B | 再利用価値が高い | 推奨 |
| C | あると便利、実害は小 | 任意 |
| D | 無関係 / 一度きり | 不要 |

#### 明文化する基準

- パターンとして再現するかで判断する (発生回数ではない)
- 1 回でもパターン化できるものは対象
- 実害があったものは 1 回でも対象

#### 明文化しない基準

- 偶然性が高い
- 自明すぎるもの
- このセッションだけの特殊状況

### ステップ 3: 昇格先の判断

| 知見の性質 | 昇格先 |
| --- | --- |
| `paths:` で対象を絞れる制約 | `.claude/rules/{topic}.md` |
| 特定タスクでのみ参照される知識 | `.claude/skills/ref-{topic}/` |
| 繰り返し実行した対話的な手順 | `.claude/skills/op-{name}/` |
| agent の振る舞い改善 | `.claude/agents/{name}.md` |
| プロジェクト概要・ドメイン知識 | `CLAUDE.md` |

既存ファイルに追記で済むなら追記。新規ファイル乱立を避ける。

### ステップ 4: ユーザーに提示

レポートを一括提示する。末尾に「対処項目サマリ」を付ける。

### ステップ 5: 対処項目を TaskCreate に登録

レポートの対処項目サマリ全件を TaskCreate で登録する。
TaskCreate が使えない環境では、対処項目サマリをそのまま進行リストとして扱い、順に処理する。

### ステップ 6: 1 項目ずつ自己検証・対話・反映

tasks に登録した各項目について、順に自己検証・対話・承認・その場で反映を行う。
AskUserQuestion ではなく自由対話形式。

まず全項目の一覧を項目名の箇条書きのみで提示する。
ユーザーが確認したら、1 件ずつ提示フォーマットに則って提示・対話に入る。
各項目の対話開始時に `[x/n] {項目名}` 形式で進捗を示す。

#### codify 前の自己検証

ルールや skill の文面をユーザーに提示する前に、以下の検証を自分で行う。
検証に通らなければ文面を修正してから提示する。
2 回修正しても通らなければその旨をユーザーに伝えて対話に入る。

##### a. 根本原因の掘り下げ

インシデントから 3 段階以上遡る:

- 何をしたか (行動)
- なぜそうしたか (判断)
- なぜそう判断したか (前提・思考パターン)

表面の行動ではなくルール化すべきは最深層の判断構造。
「X をした → X するな」のような行動のそのままのルール化は、形が変わると適用できない。

##### b. 適用シミュレーション

ルールを書いたら、元のインシデントの場面を具体的に再現する:

- 「自分がまさにあの行動をしようとしている瞬間に、このルールを読んだら止まるか？」
- 止まらないなら、なぜ止まらないかを分析してルールを修正する

##### c. スコープチェック

- 狭すぎないか: このインシデントの具体形にしか適用できないルールになっていないか。インシデント固有の名前 (ファイル名、ツール名、シナリオ名) がルール本文に残っていたら狭い
- 広すぎないか: 何にでも当てはまって判断基準にならないルールになっていないか。誰でも同意するが行動を変える基準にならないなら広い
- 直前のインシデントに引っ張られていないか: 補足説明がインシデントの再説明になっていないか

##### d. 冷読テスト

このセッションのコンテキストを知らない人がこのルールを読んで「何をすればいいか」がわかるか。
インシデント固有の文脈に依存した表現になっていないか。

#### 提示フォーマット

自己検証を通過したら、以下の構造でユーザーに提示する。

- 何が起きたか: セッション中の具体的な事実 (行動と結果)
- 本来どうあるべきだったか: 理想の判断・行動
- なぜ失敗したか: 根本原因 (自己検証 a で掘り下げた判断構造)
- 提案: 反映先ファイルパスと変更内容のドラフト

「何が起きたか」「本来どうあるべきだったか」「なぜ失敗したか」は提案の判断材料としてユーザーに見せるもので、ルール本文には含めない。

### ステップ 7: 変更の確定

全項目の対話・反映が完了したら、変更の確定方法をユーザーに確認する。
コミット、PR 作成、保留など、プロジェクトのワークフローに合わせる。

## 注意事項

- 勝手に書き換えない。ユーザー承認を必ず経る
- 過剰に抽出しない。「とりあえず codify」はノイズ化の原因
- 承認された項目は関連する変更を漏れなく反映する
