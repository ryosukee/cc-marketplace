# pre-impl-interview

コード実装の前段として、ユーザーへの質問を徹底し、曖昧さのない仕様書を作る。
plan 作成・implement 実行の入力に使う。

## 横断原則: プロジェクト方針への適応

この skill はプロジェクト非依存だが、使われるプロジェクトに fit する。
全フェーズを通じて以下を守る:

- Phase 0 で CLAUDE.md / rules / ドキュメント規約を読み取り、以降の全フェーズで参照する
- 調査はプロジェクトで利用可能なツールを優先する (Serena MCP, LSP 等があれば使う。なければ Read / Grep / Glob / Agent で代替)
- 質問の粒度・観点はプロジェクトのドメインとアーキテクチャに合わせる
- 出力ファイルの配置先・命名・フォーマットはプロジェクトの規約に従う
- plan テンプレートや仕様書テンプレートがあれば、それに合わせる
- プロジェクトの用語を使う。プロジェクト内で確立された概念は skill 側で再定義しない

「特定プロジェクトの仕組みを知っている」のではなく、「目の前のプロジェクトを読んで合わせる」。

## Phase 0: プロジェクト文脈の把握

他のフェーズに先立ち、プロジェクトの方針を理解する。

### 読み取る対象

1. `CLAUDE.md` (概要、技術スタック、ディレクトリ構成、コーディング規約、ワークフロー)
2. `.claude/rules/` (プロジェクト固有のルール。paths 付きルールは対象範囲を確認)
3. plan テンプレート・仕様書テンプレートの有無 (`docs/`, `plans/`, `specs/` 等を Glob で探索)
4. 利用可能な MCP サーバー (Serena 等のコード解析ツールがあるか)
5. 既存の plan ファイルや仕様書があれば、そのフォーマットとトーンを確認

### 把握した内容の活用

- Phase 2 の調査方法の選択に使う
- Phase 3 の質問の観点・粒度に使う
- Phase 4 の出力フォーマット・配置先に使う

Phase 0 の結果をユーザーに報告する必要はない。内部的に保持して以降で参照する。

## Phase 1: 入力の理解

引数を解釈する。

- ファイルパスが渡された場合: そのファイルを Read して概要として取り込む
- テキストが渡された場合: そのまま概要として保持する
- 引数なしの場合: AskUserQuestion で「何を実装したいですか?」と聞く
  (選択肢ではなく自由入力を前提とする。ただし直近の会話文脈からドメインが推定できる場合は、候補を選択肢として提示してもよい)

取り込んだ概要から、実装の対象範囲を推定する。
この時点では推定にとどめ、確定は Phase 3 で行う。

## Phase 2: コードベース調査

ユーザーに質問する前に、自分で調べられることは調べる。
「コードを読めばわかること」を質問するのは skill の品質に反する。

### 調査ツールの選択

Phase 0 で把握したプロジェクトの利用可能ツールに応じて調査方法を選ぶ。

- Serena MCP がある場合: シンボル解決・型情報・依存グラフを優先的に活用する
- LSP がある場合: 定義ジャンプ・参照検索を活用する
- 上記がない場合: Agent (Explore) を起動して網羅的に調査する

いずれの場合も Read / Grep / Glob は補助的に使える。
プロジェクトのツールスタックで最も効率的な調査手段を選ぶ。

### 調査対象

Phase 1 の概要に基づいて以下を調査する:

1. 対象となるコードの構造 (ディレクトリ、主要ファイル、エントリポイント)
2. 関連する既存実装 (類似機能、関連モジュール、共通パターン)
3. 依存関係 (import graph, 共有リソース、外部ライブラリ)
4. テスト構成 (テストファイルの配置、テストフレームワーク、カバレッジ方針)
5. 既に実装済みの部分 (重複実装の回避。部分的に存在する場合は範囲を特定)
6. プロジェクト固有の制約 (CLAUDE.md や rules に書かれている制約が対象領域に影響するか)

### 調査結果の報告

調査フェーズの結果をユーザーにテキストで報告する。以下を含める:

- 対象領域の現状 (何がどこにあるか。ファイルパスと行番号を添える)
- 既に実装済みの部分 (あれば)
- 関連する既存パターン (プロジェクト内で類似の実装がどうなっているか)
- 質問が必要な未確定事項の予告 (「以下の点についてこれから質問します」)

## Phase 3: インタビュー

AskUserQuestion を使って、仕様の曖昧さを解消する。
質問の品質がこの skill の核心。

### 質問の原則

- 自明な質問をしない。Phase 2 の調査で判明したことは聞かない
- 上流の判断を先に確認してから下流の詳細を聞く
- AskUserQuestion の選択肢形式を活用する (自由記述より選択肢の方が回答しやすい)
- 独立した質問は 4 問までまとめて聞く (AskUserQuestion の上限)
- 5 問以上ある場合は複数回に分ける。分ける場合は事前に「全 N 問を M 回に分けて確認します」と提示する
- 選択肢を出す前に、判断の上流にある構造的決定を先に明示する
- 選択肢の description には十分な情報を入れる。トレードオフ、影響範囲、具体例を含める

### 質問のカテゴリ

以下の観点から質問を構成する。Phase 2 で判明済みの観点はスキップする。
プロジェクトのドメインに応じて観点の重み付けを変える。

1. 仕様の曖昧さ: 正常系の動作、エッジケース、エラー時の挙動
2. 設計選択肢: 複数のアプローチがある場合のトレードオフと判断
3. UI/UX: 表示、操作、レスポンス (UI がある場合)
4. 既存機能との整合性: 影響範囲、breaking change の許容
5. 優先度とスコープ: MVP で何を入れるか、後回しにするものは何か
6. テスト方針: 何をどのレベルでテストするか
7. パフォーマンス・スケーラビリティ: 想定データ量、レスポンス要件 (該当する場合)
8. セキュリティ: 認証・認可・入力バリデーション (該当する場合)

### インタビューの進め方

1. まず最も影響の大きい上流の判断 (アーキテクチャ選択、スコープの確定) を聞く
2. 上流の回答を踏まえて、下流の詳細 (具体的な仕様、エッジケース) を聞く
3. 各ラウンドの質問を送る前に、前の回答を踏まえて次に何を聞くか判断する
4. 必要に応じて追加のコードベース調査を挟む (回答で言及された未調査の領域がある場合)
5. 曖昧さがなくなったと判断したら終了を宣言する

### 終了条件

以下の全てを満たしたらインタビュー終了:

- 実装対象の範囲が明確
- 正常系の動作仕様に曖昧さがない
- 設計選択肢について判断が確定している
- エッジケース・エラーケースの扱いが決まっている (少なくとも方針レベルで)

終了を判断するのは skill 側。「まだ質問はありますか?」とは聞かない。
曖昧さが残っていれば質問し、なければ終了を宣言して Phase 4 に進む。

終了宣言時には、確定した仕様のサマリーを箇条書きで提示してからファイル出力に移る。

## Phase 4: 仕様書の出力

インタビュー結果をまとめた仕様書を Markdown ファイルとして出力する。

### 出力先の決定

プロジェクトの規約に従う:

1. プロジェクトに plan / spec 用のディレクトリがあればそこに配置
2. 既存の仕様書・plan ファイルがあればその隣に配置
3. どちらもなければ `docs/` に配置 (`docs/` もなければ作成してよいか確認)

ファイル名は内容を表す kebab-case。
プロジェクトに命名規約があればそれに従う。
なければ `spec-{feature-name}.md` の形式。

### 仕様書の構成

プロジェクトにテンプレートがあればそれに従う。なければ以下の構成で書く:

1. 概要: 何を実装するか (1-2 文)
2. 背景: なぜ必要か
3. スコープ: 含むもの / 含まないもの
4. 仕様:
    - 機能要件 (動作仕様)
    - 非機能要件 (あれば)
5. 設計判断: インタビューで確定した選択とその理由
6. 影響範囲: 変更が及ぶ既存コード・機能 (Phase 2 の調査結果を反映)
7. テスト方針: 何をどうテストするか
8. 未決事項: インタビューで保留にしたもの (あれば)

仕様書の文体・トーンもプロジェクトの既存ドキュメントに合わせる。

### 出力後

仕様書ファイルのパスをユーザーに伝える。
この仕様書を plan skill や implement の入力として使えることを案内する。

## 禁止事項

- 推測で仕様を埋めない。不明な点は必ずユーザーに質問する
- コードの変更をしない。この skill の責務は仕様の明確化まで
- 「大体これでいいでしょう」と質問を省略しない
- プロジェクトの規約を無視しない。Phase 0 で把握した方針に従う
- AskUserQuestion を使わずにテキストだけで質問しない (選択肢形式を活用する)
