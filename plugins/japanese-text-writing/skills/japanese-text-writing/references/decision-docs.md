# 判断文書の規範

対象: 設計書、ADR、RFC、提案書、技術調査・検証レポート、障害報告・進捗報告、PR 説明、
Issue・コミットメッセージ・チャットでの連絡、ターミナルでの報告・返答。
読者はこれを根拠に判断するか行動する。結論と根拠の距離を最短にする。

## 構成

- 結論・推奨を先頭に置く。「A を推奨する。理由は 2 点」から始め、調査の経緯は後ろへ送る
- 冒頭に地図を置く: 目的 / 決めること / 読み方
- 選択肢の比較は観点を揃える（列 = 選択肢、行 = 観点）。決め手の観点を本文で名指しする
- 推奨には根拠を併記し、推奨以外も選べること（覆せること）を明示する
- 調査は方法と結果を分けて書き、再現条件（環境・バージョン・手順）を残す

## 事実の扱い

- 事実と解釈を分ける。「ログに X が出ている（事実）。原因は Y と考える（解釈）」の形で書く
- 数値には実測か推定かを付け、出典・エビデンスへのリンクを添える
- 確定していないことを確定として書かない。未確認の項目は未確認と明示して列挙する
- 判断の前提にした制約（期限・リソース・既存構成）を本文に残す。前提が変われば結論が変わるため

## 報告の短縮形

報告・返答は判断文書の圧縮版として書く。

- 最重要の一文（何が起きたか・何が分かったか・何をしたか）を最初に置く
- 次のアクション（誰が・何を・いつまでに）を明示して終える
- 長い経緯は参照先へ逃し、本文は読者が次の行動を決めるのに必要な分だけにする
- 失敗・未完了を成功と紛れる書き方をしない。結果は結果のまま書く
