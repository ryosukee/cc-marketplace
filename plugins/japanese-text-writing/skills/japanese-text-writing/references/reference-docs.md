# 参照ドキュメントの規範

対象: README、API リファレンス、設定ガイド、運用手順、セットアップ手順。
読者はこれを通読せず「使う」。必要な箇所へ最短で到達でき、読んだとおりに実行できることを最優先する。

## 構造

- 冒頭に地図を置く: このドキュメントが何で、誰が、いつ使うものかを 3 行以内で書く
- 見出しだけを拾って目的の節に到達できる構成にする。見出しは内容を特定できる具体的な句にする
- 1 節 1 トピック。同じ情報を複数の節に書かない（変更時に矛盾の原因になる）
- 比較・対応関係は表、並列の列挙は箇条書き、理由や因果の説明だけを地の文にする
- 表のセルは流し読みできる短い判定値にし、文を書かない。詳細は表の直下の補足に出す

## 手順

- 1 手順 1 操作。1 つの番号の中で 2 つの操作をさせない
- 手順の前に前提（必要な権限・環境・バージョン）、後に完了確認（何が確認できれば成功か）を書く
- コマンド・設定値は実行して確認済みのものを書く。未検証なら未検証と明示する
- 環境による分岐（OS 別など）は手順の途中で分岐させず、節を分ける

## 関心事の分離

- 書く前に「この記述は誰が・いつ読むものか」を問い、文書の関心事に合う内容だけを書く
- 恒常的な運用 rule・手順書には運用の規範だけを書く。一度きりのセットアップ手順、環境固有の事情、
  導入の経緯や日付は、環境側の文書（dotfiles の docs 等）や設計記録に分離する
- 判断基準: その記述が変わるタイミングは、この文書が変わるタイミングと同じか。別なら関心事も別
- 汎用 rule に固有事例を書くのは why の実例として最小限に留め、例と分かる形にする

why: 運用 rule の改訂時に、rule の関心事ではないセットアップ手順・パッケージ管理の事情・移行日を
書き込み、複数回の差し戻しを受けた実例（2026-07。前セッションでも同種の混入が再発）。
「この rule を読めば必要な情報が全部揃うべき」という自己完結バイアスが原因で、
環境固有の情報を環境側の文書へ分離して解決した。

## 文体

- だ体で書く。状態の表記は体言止めでよい（「デフォルト: 無効」）。
  手順の指示文はですます調か命令形のどちらかに文書内で統一する
- 読者を「あなた」と呼ばず、必要なら役割名（利用者・管理者）で書く
