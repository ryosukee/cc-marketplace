# session-retrospective

セッション末尾の振り返り skill。以下を行う:

1. セッションで得られた学びを `.claude/` 配下や `CLAUDE.md` に昇格する
2. 既存の rules/skills/agents の陳腐化をチェックする
3. セッション内で話題に上がったが実現しなかったことを検出する
4. 次にやるといいことを提案する

## 手順

### 1. セッションの棚卸し

時系列ではなくテーマごとに整理する。

- 何が繰り返し起きたか
- 何が実害になったか (バグ、レビュー差し戻し、やり直し、事故)
- どんな判断ルールが発生したか
- どんなワークフローが機能した / しなかったか

### 2. 学びの抽出と優先度分類

棚卸しの結果から、明文化すべき候補を抽出し優先度を付ける。

| 優先度 | 基準 | 対応 |
| --- | --- | --- |
| A | 実害があった + 将来同じパターンが再現する | 強く推奨 |
| B | 再利用価値が高い (複数ケースで有効、再現の予兆あり) | 推奨 |
| C | あると便利、実害は小 | 任意 |
| D | 無関係 / 一度きり / 偶発 | 不要 |

各候補は以下の構造で記述する:

```markdown
#### {候補名} [優先度]

- 発見: 何が起きたか / 何がわかったか
- 根拠: なぜそう言えるか (実害の内容、再現条件、具体例)
- アクション: どこに何を書くか (追記先 or 新規作成先)
```

#### 明文化する基準

- **パターンとして再現するようなものか** で判断する (今回発生した回数ではない)
- 1 回しか遭遇していなくてもパターン化できるものは対象 (例: "experimental addon は peer dep 衝突を起こす")
- 実害があったものは 1 回でも対象
- 同じ判断を 2 回以上したものは対象

#### 明文化しない基準

- 偶然性が高い (毎回違う原因で同じ症状)。根本原因が分かってから codify
- 自明すぎるもの
- このセッションだけの特殊状況
- 個人の好みで一般化できないもの

### 3. 既存ドキュメントとの照合 + 陳腐化チェック

`.claude/rules/`, `.claude/skills/`, `.claude/agents/`, `CLAUDE.md` をざっと確認し:

- 候補のうち既に記載済みのものを除外する (未記載の知識だけを残す)
- セッション中に使った rules/skills が実態と乖離していないか点検する:
  - 手順・フォーマット・チェックリストが実際の運用と合っているか
  - ユーザーから受けた指摘・訂正のうち、ルール化されていないものはないか
  - 使われていないセクション・機能はないか (撤去 or 簡素化の候補)

陳腐化が見つかった場合は学びの候補リストに追加する。

auto memory の棚卸しも行う。memory に蓄積された知見のうち、
rule や skill に昇格すべきものがないかチェックする。
memory は揮発性の一時メモであり、
持続的に参照すべき知見は rule/skill に昇格させる。

### 4. 昇格先の判断

| 知見の性質 | 昇格先 |
| --- | --- |
| `paths:` で対象を絞れる制約 | `.claude/rules/{topic}.md` |
| 特定タスクでのみ参照される知識 | `.claude/skills/ref-{topic}/` |
| 繰り返し実行した対話的な手順 | `.claude/skills/op-{name}/` (task skill) |
| プロジェクト概要・ドメイン知識の更新 | `CLAUDE.md` |

- 既存ファイルに追記で済むなら追記する。新規ファイル乱立を避ける
- 独立したトピックで既存ファイルに入れるのが不自然なら新規作成

### 5. 表現強化の注意

既存ルールの文言を強化する形で対応する場合:

- 曖昧な表現を「必須」「禁止」に強めると遵守率は上がる
- ただし「強化しただけで解決した気になる」罠に注意する
- 強化は対処療法。root cause (判断基準の不足、ワークフローの欠陥、監視の不在) が残っていれば再発する
- 強化を検討したら「強化で足りるか / 構造変更が必要か」を併せて考える

### 6. やり残し検出

セッション中の会話を振り返り、以下に該当するものをリストアップし、ユーザーに提示する

- やろうと思っていたが着手しなかったこと
- ユーザーから要求があったが完了しなかったこと
- 議論の中で出た改善案で実行されなかったもの
- 途中で方針変更して取りやめたが、元の案にも価値があるもの

リストアップだけ行い、棚卸し (JIRA 化、TODO 化等) はしない。ユーザーが見て判断する材料を提示する。

### 7. 次アクション提案

今回のセッションを踏まえて、次にやるといいことを提案する:

- 実行したタスクに対する改善案 (「ここをこうするともっと良くなる」)
- 今回の作業から派生する自然な次ステップ
- セッション中に見えた課題で、別セッションで取り組むべきもの

押し付けではなく提案。ユーザーが採用するかは任意。

### 8. ユーザーに提示

まず以下のフォーマットでレポート全体を一括提示する。
その後、判断が必要な項目 (A/B 候補 + 陳腐化) が多い場合は
AskUserQuestion で 1 項目ずつ確認を取る。
少なければ一括承認でよい。

```markdown
# セッション振り返り

## 発生した出来事 (テーマ別)
1. {テーマ} — 何が起きたか、実害、関連ファイル
2. ...

## 昇格候補リスト

### A (強く推奨)
#### {候補名}
- 発見: ...
- 根拠: ...
- アクション: ...

### B (推奨)
...

### C (任意)
...

## 陳腐化の指摘
- {ファイル}: {何が古いか、どう直すか}

## やり残し
- {未完了項目}: {何をやろうとしていたか}

## 次アクション提案
- {提案}: {なぜやるといいか}

---
A と B を反映していいですか?
```

### 9. 承認と反映

- ユーザーが承認した項目のみ該当ファイルに追記・新規作成する。勝手に書かない
- 陳腐化の修正もユーザー承認後に行う
- コミットするかどうかもユーザーに確認する
- コミットする場合は全変更を 1 コミットにまとめる

## 注意事項

- 勝手に書き換えない。ユーザー承認を必ず経る
- 過剰に抽出しない。「とりあえず codify」はノイズ化の原因
- 1 commit で完結する。複数ファイル更新を 1 コミットにまとめて commit history を綺麗に保つ
