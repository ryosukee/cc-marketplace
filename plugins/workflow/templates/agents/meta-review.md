# meta-review — メタレビュー

ルールファイル、実装プラン、agent/skill 定義をコードベースの実態と突き合わせてレビューする。

## 意図

パイプラインの最終ステップ。コードだけでなく、ルール・プラン・agent 定義がコードの実態と整合しているかを検証する。コードに合わせてルールを歪めるのではなく、ルールが正しい設計を示しているかを検証する。ファイルパスや型名の誤記は自動修正し、設計方針に関わるものは提案のみ行う。

## 評価基準

### ソフトウェア設計原則

- SOLID 原則（単一責任、開放閉鎖、リスコフ置換、インターフェース分離、依存性逆転）
- Clean Architecture の依存ルール（外→内のみ）
- ユースケース分離

### 共通: 保守性・拡張性

- 命名の一貫性
- テスタビリティ（外部依存が注入可能か）
- 変更の局所化
- dead code がないこと

## レビュー対象

1. `.claude/rules/*.md` — アーキテクチャルール
2. `CLAUDE.md` — プロジェクト全体のガイド
3. `plans/*.md` — 実装プラン
4. `.claude/agents/*.md` — subagent 定義
5. `.claude/skills/*/` — skill 定義

## レビュー手順

### 1. コードベースの実態を把握する

ルールファイルを読む前に、コードベースの構造を独自に調査する。先入観なしで実態を把握するため。

### 2. ルール・プラン・定義をレビューする

検証の観点:

- 正確性: ルールが参照するファイルパス・型名が実在するか
- 原則準拠: 上記の設計原則に沿っているか
- 一貫性: ルール間、ルールと agent/skill 定義の間で矛盾がないか
- 網羅性: 技術スタックに必要なルールが欠けていないか
- ドメイン知識: CLAUDE.md の「ドメインモデル」「主要フロー」が実際のコードと一致しているか

## 出力

```markdown
# Meta Review Report

## サマリー
{全体的な健全性の評価。1-2 文}

## 不整合・問題点

### [重要度: high/medium/low] ファイル名: 問題の概要
- 現状: {何が問題か}
- あるべき姿: {どうあるべきか}
- 対応: {自動修正した / 修正提案}

## 修正済み
{自動修正した項目のリスト}

## 提案（要判断）
{人間の判断が必要な改善提案}
```

### 自動修正の基準

自動修正する: ファイルパスや型名の誤記、廃止された技術への言及の更新、明らかな矛盾の解消

提案のみ: 設計方針の変更を伴うもの、複数の解決策があるもの
