# ワークフロー同期チェック

registry.json に登録されたリポジトリから最新の `.claude/` 構成を取得し、テンプレートとの差分を検出・提案する。

## 意図

workflow plugin のテンプレートは実プロジェクトから抽出したもの。プロジェクト側で agent/skill/rules が改善されたらテンプレートに取り込みたい。この skill がその同期を担う。

テンプレート→プロジェクト方向の同期は `/workflow:setup` が担当する（setup は新規セットアップだけでなく、既存プロジェクトへの差分補完も行う）。

## 前提

- registry: `${CLAUDE_SKILL_DIR}/../../registry.json`
- テンプレート: `${CLAUDE_SKILL_DIR}/../../templates/`

## ワークフロー

### ステップ 1: registry の読み込み

`registry.json` を読み、登録リポジトリの一覧を取得する。

### ステップ 2: リポジトリの .claude/ を取得

各リポジトリについて、以下の順でアクセスを試みる:

1. `ghq list --full-path` で `{github}` に一致するパスを探す
2. 見つかればローカルから読む
3. 見つからなければ `gh api` で GitHub から取得する（ファイル一覧 → 各ファイルの内容）
4. GitHub API も失敗した場合（認証エラー、リポジトリ不在、レート制限等）→ そのリポジトリをスキップし、理由を記録する

取得対象:
- `.claude/agents/*.md`
- `.claude/skills/*/SKILL.md` (または `skill.md`)
- `.claude/rules/*.md`

#### 構成バリデーション

取得できた��ポジトリについて、ワークフロー構成として有効かを検証する。以下の全てを満たさない場合はその���ポジトリをスキップする:

- `.claude/agents/` に 1 つ以上の agent 定義 (`.md`) がある
- `.claude/skills/` に 1 つ以上の skill 定義がある

スキップ時は理由を記録し、ステップ 4 のレポートで報告する。全リポジトリがスキップされた場合は、スキップ理由の一覧を表示して終了する。

### ステップ 3: 差分の検出

テンプレートとリポジトリの `.claude/` を比較する。

#### A. テンプレートへの取り込み候補

リポジトリ側の改善で、テンプレートに反映すべきもの:

- リポジトリにあるがテンプレートにない agent/skill（新規追加の候補）
- 両方にあるが、リポジトリ側に汎用的な改善がある（手順の追加、ルールの改善、allowedTools の変更等）
- メタルール (rule-authoring.md, claudemd-authoring.md) の改善

判断基準:
- リポジトリ側の変更が tech stack 非依存（汎用的）→ 取り込み候補
- リポジトリ側の変更がプロジェクト固有 → 取り込まない

#### B. リポジトリへの反映推奨

テンプレート側の改善で、リポジトリに反映を推奨するもの:

- テンプレートにあるがリポジトリにない agent/skill
- テンプレート側に新しいセクション・手順がある
- テンプレートの「意図」セクションがリポジトリにない

#### C. アーキテクチャ rules の構造チェック

プロジェクト固有の rules の内容は比較対象外だが、ルールの書き方・構造にベストプラクティスの改善があれば報告する（frontmatter の `paths` 設定、セクション構成等）。

### ステップ 4: 同期レポートの出力

```markdown
# Workflow Sync Report

## {リポジトリ名} ({owner/repo})

### テンプレートへの取り込み候補

#### [{ファイル種別}] {ファイル名}: {変更の概要}
- 差分: {何がどう変わったか}
- 汎用性: {なぜテンプレートに取り込むべきか}
- 提案: {具体的な変更内容}

### リポジトリへの反映推奨

#### [{ファイル種別}] {ファイル名}: {変更の概要}
- 差分: {何がどう変わったか}
- 提案: {具体的な変更内容}

### 同期済み
差分のないファイル一覧。

### スキップされたリポジトリ
(あれば) リポジトリ名と理由。
```

### ステップ 5: 反映の実行

レポートを表示後、AskUserQuestion で反映するかを確認する。各提案は個別に承認/スキップ可能。

- テンプレートへの取り込み → workflow plugin のテ��プレートファイルを更新
- リポジトリへの反映推奨 → 対象リポジトリの `.claude/` ファイルを更新（ローカルクローンがある場合のみ）

## registry.json の構造

```json
{
  "repositories": [
    {
      "name": "表示名",
      "github": "owner/repo",
      "description": "リポジトリの説明",
      "role": "primary | reference",
      "note": "補足情報"
    }
  ]
}
```

- `role: primary` — テンプレートの原型となるリポジトリ。双方向の同期対象
- `role: reference` — 参考にするリポジトリ。テンプレートへの取り込みのみ（リポジトリへの反映推奨は行わない）

## 注意

- リポジトリの `.claude/` 内の tech stack 固有部分（Go のビルドコマンド等）は差分として報告しない。テンプレートの placeholder に対応する部分は比較対象外
- 差分の判断は内容ベース。ファイルのタイムスタンプは使わない
- GitHub API のレート制限に注意。ローカルクローンがあればそちらを優先する
