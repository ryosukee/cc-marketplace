# test-plan

要件定義書の受入基準と設計書のコントラクトを入力に、テスト計画を独立した文書として作成する。
受入基準を全件テストに変換し、漏れをゼロにすることを目的とする。
トレーサビリティ表で受入基準とテストの対応を可視化し、漏れを構造的に防ぐ。

## テスト設計の原則: 入り口と結果の網羅

テスト計画で最優先に網羅すべきはエントリポイント (入り口) から見た E2E 的な観点。
シーケンス図をイメージして、ユーザーがアクセスする入り口の全ケースと、それに対する結果を網羅する。

バックエンド: API endpoint が入り口。各エンドポイントに対して、リクエストのバリエーション (正常系・異常系・境界値) と期待するレスポンス・副作用を網羅する。

フロントエンド: ユーザー操作が入り口。Testing Library の哲学と同じく、ユーザーが見て操作するものを起点に、操作の結果として画面に何が起きるかを網羅する。内部の state や hook を直接テストするのではなく、ユーザー操作 → 結果の流れで検証する。

この E2E 的な観点でのテストは全プロジェクトで必須。内部状態や内部関数の単体テストはプロジェクトの方針に依存するため、Phase 0 で把握した規約に従う。

## 横断原則: 上流文書へのフィードバック

要件定義書と設計書は所与の入力ではない。
テスト計画を立てる過程で上流文書の不備が見つかることは自然なことで、むしろテスト計画の価値の一つ。

テスト観点から発見しやすい上流の問題:

- 受入基準の網羅性不足 (テストに変換しようとして初めて「この振る舞いの基準がない」と気づく)
- 受入基準の曖昧さ (テストのアサーションを書こうとして「何をもって成功とするか不明」と気づく)
- 設計書のコントラクト不足 (テスト対象を特定しようとして「このインターフェースの振る舞いが未定義」と気づく)
- 要件と設計の不整合 (受入基準が設計のコントラクトでカバーされていない)

発見した場合の対応:

1. テスト計画書に「上流文書へのフィードバック」セクションを設け、発見事項を記録する
2. フィードバック内容をユーザーに報告し、上流文書の修正が必要か確認する
3. ユーザーが修正を承認した場合、該当する上流文書を直接更新する (要件定義書・設計書とも)
4. テスト計画書は更新後の上流文書に基づいて最終化する

上流文書の修正は test-plan skill のスコープ内。requirements / design skill に差し戻す必要はない。

## 横断原則: プロジェクト方針への適応

この skill はプロジェクト非依存だが、使われるプロジェクトに fit する。
全フェーズを通じて以下を守る:

- Phase 0 で CLAUDE.md / rules / テスト規約を読み取り、以降の全フェーズで参照する
- 調査はプロジェクトで利用可能なツールを優先する (Serena MCP, LSP 等があれば使う。なければ Read / Grep / Glob / Agent で代替)
- テストの命名規約・配置規約・フレームワーク選択はプロジェクトの慣行に合わせる
- 出力ファイルの配置先・命名・フォーマットはプロジェクトの規約に従う
- プロジェクトの用語を使う

## Phase 0: プロジェクト文脈の把握

他のフェーズに先立ち、プロジェクトのテスト方針を理解する。

### 読み取る対象

1. `CLAUDE.md` (技術スタック、テストフレームワーク、ディレクトリ構成)
2. `.claude/rules/` (テストに関するルール。最も信頼度の高いソース)
3. プロジェクト内のドキュメント: テスト方針、CI 設定、既存のテスト計画等を Glob で探索する
4. 利用可能な MCP サーバー

### テスト規約の把握

以下をプロジェクトから読み取る。明示されていない項目は Phase 2 の調査で推定する:

- テストフレームワーク (Go: Ginkgo/testing、JS: Vitest/Jest、E2E: Playwright/Cypress 等)
- テストファイルの配置規約 (コロケーション、`__tests__/`、`tests/` 等)
- テスト命名規約 (BDD describe/it、AAA arrange/act/assert 等)
- モック・フィクスチャの慣行
- E2E テストのパターン (ページオブジェクト、web-first assertion 等)
- CI でのテスト実行構成

Phase 0 の結果をユーザーに報告する必要はない。内部的に保持して以降で参照する。

## Phase 1: 入力の読み込み

設計書と要件定義書を読み込む。

### 設計書の読み込み

- ファイルパスが渡された場合: そのファイルを Read
- 引数なしの場合: AskUserQuestion でファイルパスを聞く

### 要件定義書の特定

設計書内の「対応する要件定義書」リンクから要件定義書を Read する。
リンクがない場合は AskUserQuestion でパスを聞く。

### 入力の検証

両文書を読み込んだら以下を確認する:

- 要件定義書に受入基準が定義されているか
- 設計書にコントラクト定義と変更対象ファイル一覧があるか

不十分な場合はユーザーに伝え、該当 skill での補完を提案する。

## Phase 2: コードベース調査

テスト計画に必要な情報を調査する。

### 調査ツールの選択

Phase 0 で把握したプロジェクトの利用可能ツールに応じて調査方法を選ぶ。

- Serena MCP がある場合: テストファイルの依存関係・テスト対象の参照を活用する
- LSP がある場合: テスト対象の定義・参照検索を活用する
- 上記がない場合: Agent (Explore) を起動して調査する

### 調査対象

1. 既存テストの網羅状況: 設計書の変更対象ファイルに対応するテストの有無と内容
2. テストパターン: プロジェクトで使われている describe/it 構造、モック方法、フィクスチャ管理
3. テストユーティリティ: 共通のヘルパー、カスタムマッチャー、テストファクトリ
4. E2E テスト基盤: ページオブジェクト、テストデータのセットアップ方法、テスト環境の構成
5. 既存テストのカバー範囲: 受入基準のうち、既存テストで既にカバーされているものの特定

### 既存テストのカバー判定

受入基準の各項目に対して、既にカバーするテストが存在するかを判定する。
判定は実際のテストコードを読んで行う。テストファイルの存在だけでは判定しない。

カバー判定の基準:

- テストが受入基準の振る舞いをアサートしている → カバー済み
- テストが存在するが受入基準の振る舞いをアサートしていない → 未カバー
- テストが存在しない → 未カバー

### 調査結果の検証

Serena MCP を優先し、なければ grep / Read で検証する。特に「既にカバー済み」の判定は、テストコードの実在とアサーション内容を確認する。

### 調査結果の報告

調査結果をユーザーにテキストで報告する:

- テストフレームワークと規約の把握結果
- 既存テストのカバー状況 (受入基準ごと)
- 新規テストが必要な受入基準の一覧
- インタビューが必要な判断事項の予告

## Phase 3: インタビュー

AskUserQuestion を使って、テスト戦略の判断を確定させる。
Phase 2 の調査で自明に決まる判断は質問せず報告にとどめる。

### 質問が必要になる場面

- テスト層の分類が曖昧な受入基準 (API テストと E2E のどちらで検証するか等)
- テスト不可能な受入基準の扱い (主観的な品質基準、外部サービス依存等)
- モック戦略の判断 (外部 API をモックするか、テスト環境を用意するか)
- テスト環境の制約 (CI で E2E が実行できるか、ブラウザ拡張のテスト手段等)
- カバレッジの優先度 (全受入基準を一度にテスト化するか、段階的に進めるか)

### 質問の原則

- Phase 2 の調査で判明した事実を根拠に選択肢を絞る
- プロジェクトの既存テストパターンに沿う選択肢を推奨として提示する
- AskUserQuestion の選択肢形式を活用する
- 独立した質問は 4 問までまとめて聞く

### 終了条件

以下の全てを満たしたらインタビュー終了:

- 全受入基準のテスト層分類が確定している
- テスト不可能な受入基準の扱いが決まっている
- モック・フィクスチャの方針が確定している
- 未確定項目がゼロ

インタビューの必要がない場合 (調査で全て確定した場合) は Phase 3 をスキップして Phase 4 に進む。スキップする旨をユーザーに報告する。

## Phase 4: テスト計画書の出力

トレーサビリティ表とテストシナリオをまとめたテスト計画書を Markdown ファイルとして出力する。
出力前に spec-reviewer agent でセルフレビューを行い、指摘があれば修正してから出力する。

### 生成アルゴリズム

#### Step 1: 受入基準の全件抽出

要件定義書の受入基準セクションから全項目を抽出する。
受入基準がリスト形式でない場合 (散文の中に検証条件が埋まっている場合) は、検証可能な振る舞いを 1 つずつ分離して列挙する。

Phase 2 の調査で「テストすべきだが受入基準にない振る舞い」を発見した場合:

1. 上流フィードバックとしてユーザーに報告する (「この振る舞いは受入基準にないがテストすべき」)
2. ユーザーが同意したら要件定義書の受入基準を更新する
3. 更新後の受入基準をトレーサビリティ表に含める

受入基準の追加なしにテスト対象を増やさない。テスト計画は受入基準が起点という原則を守りつつ、受入基準自体の不備は上流フィードバックで補完する。

#### Step 2: テスト層への分類

各受入基準を振る舞いの性質に基づいてテスト層に分類する。
分類基準はプロジェクトのテストフレームワークに合わせる。

分類の優先順位:

1. まず全受入基準に対して E2E 的なテスト (入り口 → 結果) を設計する。これが必須層
2. その上で、E2E だけでは検証が不十分な振る舞いに対して内部テストを追加する

E2E 的テストの入り口:

- バックエンド: API endpoint へのリクエスト → レスポンス + 副作用 (DB 状態、外部呼び出し等)
- フロントエンド: ユーザー操作 (クリック、入力、ナビゲーション) → 画面に表示される結果

内部テストの追加が有効な場面 (プロジェクトの方針に依存):

- 複雑なビジネスロジック (計算、バリデーション) を入り口テストだけでは組み合わせ爆発する場合
- エラーハンドリングの分岐が多く、入り口テストで全パスを通すのが困難な場合
- パフォーマンスクリティカルな処理を隔離してベンチマークしたい場合

1 つの受入基準が複数層にまたがる場合は各層にテストを配置する。

#### Step 3: 既存カバレッジの反映

Phase 2 で特定した既存テストのカバー状況をトレーサビリティ表に反映する。
カバー済みの項目は新規テスト設計の対象から除外するが、トレーサビリティ表には記載する。

#### Step 4: テストシナリオの設計

未カバーの受入基準に対してテストシナリオを設計する。
入り口 → 結果の E2E 的テストを先に設計し、必要に応じて内部テストを追加する。

バックエンド API テストの設計では:

- 各エンドポイントに対して、入力のバリエーションを網羅する (正常系、異常系、境界値、認証・認可)
- 各入力に対する期待結果を明示する (ステータスコード、レスポンスボディ、DB 状態変更、外部呼び出し)
- テストデータのセットアップ方法を明示する

フロントエンドテストの設計では:

- ユーザー操作を起点にする。内部の state や hook ではなく、ユーザーが見て触れるものでテストする
- 操作の自然な流れ (操作 → フィードバック → 結果) を 1 テストにまとめる
- 非同期処理が挟まっていても、ユーザーが追加操作なしで結果を見られるなら分割しない
- モック・フィクスチャの要件を明示する
- プロジェクトの規約に従うアサーション方式を使う

ブラウザ E2E テスト (Playwright 等) の設計では:

- ページ遷移・復元・複数画面にまたがる操作フローを検証する
- フロントエンド単体テストではカバーできないブラウザ統合の振る舞いに限定する

#### Step 5: トレーサビリティの検証

全受入基準に少なくとも 1 つのテストが紐付いていることを確認する。
紐付かない受入基準がある場合:

- テスト可能だが漏れていた → テストを追加設計する
- テスト不可能 (主観的品質基準等) → 理由を付記して除外する

### セルフレビュー

ファイルに書き出す前に、Agent ツールで `spec-reviewer` を起動する。

- subagent_type: `impl-spec:spec-reviewer`
- 入力: テスト計画書のドラフト全文、doc_type は `test-plan`、対応する要件定義書と設計書のパス
- 出力: 指摘リスト

指摘への対応は種類によって分ける:

- skill が自力で修正できるもの (トレーサビリティの記載漏れ、テスト ID の不整合等): そのまま修正する
- ユーザーへの再質問が必要なもの (テスト層の分類判断、テスト不可能な項目の扱い等): Phase 3 に戻り AskUserQuestion で確認してから修正する

修正後は再度 spec-reviewer を回す。指摘がゼロになるか、最大 5 回に達したら終了する。

### 出力先の決定

プロジェクトの規約に従う:

1. テスト計画・仕様書用のディレクトリがあればそこに配置
2. 設計書と同じディレクトリに配置 (対になる文書として)
3. どちらもなければ `docs/` に配置

ファイル名はプロジェクト規約に従う。なければ `test-plan-{feature-name}.md` の形式。
対応する要件定義書・設計書との関連がわかる命名にする。

### テスト計画書の構成

プロジェクトにテンプレートがあればそれに従う。なければ以下の構成:

1. 概要: 何の機能のテスト計画か (1-2 文)
2. 対応する文書: 要件定義書と設計書へのリンク
3. トレーサビリティ表: 受入基準とテストの対応。全受入基準を列挙し、テスト層・テスト ID・状態 (新規/既存カバー/テスト対象外) を記載
4. テストシナリオ: テスト層ごとにグルーピングし、各テストの詳細を記載
    - テスト ID
    - ファイル配置先
    - 対応する受入基準
    - テスト構造 (describe/it 等)
    - セットアップ (テストデータ、モック、フィクスチャ)
    - 操作手順 (E2E の場合)
    - アサーション対象
5. 既存テストへの影響: 変更・削除が必要な既存テストの一覧
6. テスト対象外の受入基準: テスト不可能な項目とその理由 (あれば)
7. 上流文書へのフィードバック: テスト計画の策定過程で発見した要件定義書・設計書の不備と、実施した修正の記録 (あれば)

### テスト計画書の品質基準

- 全受入基準がトレーサビリティ表に含まれている (漏れゼロ)
- 各テストシナリオのアサーションが具体的で検証可能
- テストファイルの配置先がプロジェクトの規約に沿っている
- モック・フィクスチャの要件が明示されている
- 既存テストとの重複がない
- 曖昧な表現 (「適切にテストする」「必要に応じて検証」) を残さない
- Phase 0 で把握したテスト規約と矛盾する記述がない

### 出力後

テスト計画書のパスをユーザーに伝え、次のステップを案内する。
案内内容はプロジェクトのワークフローに合わせる (手動実装、team-implement への引き渡し等)。

```
テスト計画書を出力しました: {path}
```

## 禁止事項

- 推測でテスト方針を決めない。プロジェクトの慣行と受入基準に基づく
- コードの変更をしない。この skill の責務はテスト計画と上流文書のフィードバックまで
- 受入基準に存在しない振る舞いを受入基準の追加なしにテスト対象に含めない。テストすべき振る舞いを発見したら、まず上流フィードバックで受入基準を追加する
- AskUserQuestion を使わずにテキストだけで質問しない
- テスト計画書にテストコードの実装を書かない。テスト構造とアサーション対象を書くが、コードは書かない
