# design

要件定義書 (requirements skill の出力) を入力に、「どう作るか」を設計レベルで明確化する。
出力は設計書。実装者 (人または implement agent) がこの文書とコードベースから迷わず着手できるレベルを目指す。

## 設計書の粒度原則

設計書は「判断の記録」であり「コードの下書き」ではない。

境界の基準: **変更した場合に他のコンポーネントとの調整が必要になるもの** は設計書に書く。そうでないものは実装者に委ねる。

### 設計書が書くもの

- コンポーネント間のインターフェースとコントラクト
- データモデルとスキーマ
- 統合点とプロトコル
- 障害モードとエラーハンドリング戦略
- パフォーマンスやセキュリティの制約
- 変更対象の全インスタンス (同一パターンでも省略せず列挙する)

### 実装者に委ねるもの

- コンポーネント内部のロジック実装
- 関数の分割やローカル変数の設計
- アルゴリズムの具体的な実装方法
- コードの書き方

### この分離の理由

実装者 (人でも AI agent でも) はコントラクトの範囲内で自身の知識を最大限に活かせる。設計書が内部実装まで規定すると、実装者の創造性が制約され、かつ設計書が実質的にコードの二重管理になる。設計書はコントラクトでコードの正しさを検証可能にし、内部実装の自由度は保つ。

## 横断原則: プロジェクト方針への適応

この skill はプロジェクト非依存だが、使われるプロジェクトに fit する。
全フェーズを通じて以下を守る:

- Phase 0 で CLAUDE.md / rules / 設計規約を読み取り、以降の全フェーズで参照する
- 調査はプロジェクトで利用可能なツールを優先する (Serena MCP, LSP 等があれば使う。なければ Read / Grep / Glob / Agent で代替)
- 設計判断はプロジェクトの既存パターン・規約を尊重する。逸脱する場合は理由を明示する
- 出力ファイルの配置先・命名・フォーマットはプロジェクトの規約に従う
- プロジェクトの用語を使う

## Phase 0: プロジェクト文脈の把握

他のフェーズに先立ち、プロジェクトの方針を理解する。

### 読み取る対象

1. `CLAUDE.md` (技術スタック、ディレクトリ構成、コーディング規約、アーキテクチャ方針)
2. `.claude/rules/` (プロジェクト固有のルール。設計に影響する制約を特定。最も信頼度の高いソース)
3. プロジェクト内のドキュメント: 設計方針、既存の仕様書・要件書・設計書・ADR 等を Glob で探索する。配置場所はプロジェクトによって異なる
4. 利用可能な MCP サーバー

Phase 0 で把握した情報は以降の全フェーズで照合基準として使う。Phase 2 の調査、Phase 4 の品質チェックで、ここで把握したルールやドキュメントと矛盾する記述がないか常に検証する。

Phase 0 の結果をユーザーに報告する必要はない。内部的に保持して以降で参照する。

## Phase 1: 要件定義書の読み込み

引数で渡された要件定義書を Read する。

- ファイルパスが渡された場合: そのファイルを Read
- 引数なしの場合: AskUserQuestion でファイルパスを聞く

要件定義書を読み込んだら以下を確認する:

- スコープが明確か
- 各要件が検証可能な記述になっているか
- 受入基準が定義されているか

不十分な場合はユーザーに伝え、requirements skill での再作成を提案する。
十分な場合は Phase 2 に進む。

### 要件の未決事項の検出

requirements から引き継いだ未決事項や、読み込み時に発見した要件の曖昧さは、Phase 3 で設計判断と併せて解決する。解決した場合は要件定義書も直接更新する。設計段階で決着できない場合のみ requirements skill での再作成を提案する。

## Phase 2: コードベース深掘り調査

requirements skill の Phase 2 より深い調査を行う。
設計判断に必要な実装詳細まで把握する。

### 調査ツールの選択

Phase 0 で把握したプロジェクトの利用可能ツールに応じて調査方法を選ぶ。

- Serena MCP がある場合: シンボル解決・型情報・依存グラフ・コールグラフを活用する
- LSP がある場合: 定義ジャンプ・参照検索・型ヒエラルキーを活用する
- 上記がない場合: Agent (Explore) を起動して網羅的に調査する

### 調査対象

要件定義書の各要件に対して、以下を調査する:

1. 変更対象となるファイル群の特定 (パスと該当箇所の行番号)
2. 既存のアーキテクチャパターン (レイヤー構造、データフロー、状態管理)
3. 類似機能の実装パターン (プロジェクト内で同種の機能がどう実装されているか)
4. 依存関係の詳細 (import graph, 共有 state, event bus 等)
5. テストの構成とパターン (既存テストの書き方、カバレッジ方針)
6. DB スキーマ・API エンドポイント (該当する場合)
7. 設計上の制約 (rules で規定されたパターン、禁止事項)

### 削除・置換対象の波及調査

既存コードの削除や置換を伴う要件では、対象コードが参照している / されている箇所を全て追跡する。関連する state、副作用、テストへの波及を把握してから設計に進む。

### 既存テストの影響分析

変更・削除対象ファイルに対応するテストを特定し、削除・修正が必要なテストを洗い出す。

### 調査結果の検証

調査結果の検証は Serena MCP を優先し、なければ grep / Read で行う。特に以下は必ず自分で裏付けを取る:

- 否定的記述 (「存在しない」「していない」「のみ」): 不在を確認する
- 固有名詞 (関数名、ファイルパス、コンポーネント名): 実在を確認する

Agent に委任した調査結果も同様に検証する。Agent は「見つけられなかった」と「存在しない」を区別しない。

Phase 0 で把握した規約・既存ドキュメントを調査の照合基準として使う。

変更対象レイヤーのエントリポイントを全件確認し、スコープに関連するものが漏れていないか検証する。起動シーケンスやバックグラウンド処理の起動パスも含める。

同じ機能を複数の技術スタックで提供する場合、各スタックの制約を特定する。

### 調査結果の報告

調査結果をユーザーにテキストで報告する:

- 変更が必要なファイル群の一覧 (パスと行番号)
- プロジェクトの既存パターン
- 設計上の選択肢の予告 (「以下の点について設計判断が必要です」)

## Phase 3: 設計インタビュー

AskUserQuestion を使って、設計の選択肢を確定させる。
要件 (何を作るか) ではなく、設計 (どう作るか) に集中する。

### 質問の原則

- プロジェクトの既存パターンに沿う選択肢を推奨として提示する
- 各選択肢のトレードオフ (実装コスト、保守性、パフォーマンス、既存コードとの一貫性) を明示する
- コードベース調査で判明した事実を根拠に選択肢を絞る
- 自明な判断 (プロジェクトのパターンに 1 つしか合致しないもの) は質問せず、報告にとどめる
- 構造的な判断 (分類、グルーピング、カテゴリ分け、適用範囲のマッピング) は skill が勝手に作らない。構造はユーザーとインクリメンタルに合意しながら組み立てる。「全部」「統一」「一律」等の方針が来た場合は、全件・全パターンを洗い出して提示し、各々が本当に含まれるか確認する。回答の適用範囲を元の文脈から拡大する場合も確認する
- 設計判断の影響が大きい場合や、既存パターンと異なるアプローチが候補に上がった場合、コミュニティや公式のトレンド・推奨を調査して判断材料にする。調査が有用かどうか自体をユーザーに聞いてもよい。WebSearch/WebFetch が利用可能な場合に限る
- AskUserQuestion の選択肢形式を活用する
- 独立した質問は 4 問までまとめて聞く

### 質問のカテゴリ (設計レベル)

以下の観点から質問を構成する。調査で確定済みの観点はスキップ。

1. 既存踏襲 vs 再設計: 対象領域を既存のパターンに合わせるか、ゼロベースで設計し直すか。最も上流の判断として最初に確認する
2. アーキテクチャ: コンポーネント構成、レイヤー配置、責務分割
3. データ設計: スキーマ変更、データ構造、状態管理方式
4. API 設計: エンドポイント、リクエスト/レスポンス形式 (該当する場合)
5. 実装アプローチ: 既存コードの拡張 vs 新規作成、リファクタリングの範囲
6. エラーハンドリング方式: 例外戦略、リトライ、フォールバック
7. マイグレーション: 既存データ・既存 API との互換性、段階的リリース (該当する場合)

### 終了条件

以下の全てを満たしたらインタビュー終了:

- 各要件に対する設計方針が確定している
- 変更対象のファイルと変更内容が特定されている
- 設計上の選択肢について判断が確定し、理由が記録されている
- 未確定項目がゼロ (2 案併記は許容しない。インタビューで決着させる)

## Phase 4: 設計書の出力

インタビュー結果をまとめた設計書を Markdown ファイルとして出力する。
出力前に spec-reviewer agent でセルフレビューを行い、指摘があれば修正してから出力する。

### セルフレビュー

ファイルに書き出す前に、Agent ツールで `spec-reviewer` を起動する。

- subagent_type: `impl-spec:spec-reviewer`
- 入力: 設計書のドラフト全文、doc_type は `design`、対応する要件定義書のパス
- 出力: 指摘リスト

指摘への対応は種類によって分ける:

- skill が自力で修正できるもの (ファイル一覧と詳細の不一致、曖昧表現の書き換え等): そのまま修正する
- ユーザーへの再質問が必要なもの: Phase 3 に戻り AskUserQuestion で確認してから修正する

以下は「ユーザーへの再質問が必要」に分類する:

- 未確定項目、網羅性の不足で判断が要るもの
- 技術的事実の検証が必要なもの: まず検証し、検証結果をユーザーに提示した上で設計判断を確認する。検証結果から「自明」に見えても、判断はユーザーに委ねる
- 要件定義書の記述間の不整合: どちらを権威的ソースとするかはユーザーに確認する
- 既存コードの挙動が設計の前提になるもの (トランザクション有無、z-index の重なり順等): コードを読んで事実を確認し、その事実と共に設計判断をユーザーに提示する

分類に迷ったら「ユーザーへの再質問が必要」側に倒す。
推測で埋めた判断が間違っていた場合のコストは、追加質問のコストより高い。

実行順序: インタビューを先に行い、確定させてから自力修正に進む。
自力修正を先にやると修正モードの慣性で設計判断も自力解決してしまうリスクがある。

修正後は再度 spec-reviewer を回す。指摘がゼロになるか、最大 5 回に達したら終了する。

### 出力先の決定

プロジェクトの規約に従う:

1. 設計書・plan 用のディレクトリがあればそこに配置
2. 要件定義書と同じディレクトリに配置 (対になる文書として)
3. どちらもなければ `docs/` に配置

ファイル名はプロジェクト規約に従う。なければ `design-{feature-name}.md` の形式。
対応する要件定義書との関連がわかる命名にする。

### 設計書の構成

プロジェクトにテンプレートがあればそれに従う。なければ以下の構成:

1. 概要: 何をどう実装するかの要約 (数行)
2. 対応する要件定義書: ファイルパスへのリンク
3. アーキテクチャ:
    - コンポーネント構成とデータフロー
    - 採用したパターンとその理由
4. 変更対象ファイル一覧:
    - ファイルパス、変更種別 (新規 / 修正 / 削除)、変更内容の概要
    - 全インスタンスを列挙する。「同様」で省略しない
5. コントラクト定義:
    - コンポーネント間のインターフェース (型シグネチャ、入出力)
    - データモデル / スキーマ変更
    - API コントラクト (該当する場合)
    - 状態遷移と各状態での振る舞い契約 (該当する場合)
6. 設計判断の記録:
    - 検討した選択肢、採用した案とその理由、却下した案とその理由
7. 実装順序: 推奨する着手順
8. リスク・懸念事項 (あれば)

### 設計書の品質基準

- 粒度原則を守る: コントラクトと統合点を具体的に書き、内部実装は書かない
- 変更対象の全インスタンスが列挙されている (省略しない)
- 設計判断の理由が記録されている
- 未確定項目がゼロ
- プロジェクトの既存パターンとの一貫性が保たれている
- 曖昧な表現 (「適切に」「必要に応じて」) を残さない
- Phase 0 で把握したルール・ドキュメントと矛盾する記述がないことを確認する
- 状態遷移を定義する場合、全状態 × 全イベントの組み合わせに対して振る舞いが定義されている。未定義の組み合わせを残さない
- コントラクトとして定義するインターフェースは、全メソッドの引数型・戻り値型・副作用の有無が明記されている
- 技術スタックが異なる複数のクライアントがある場合、各クライアントのコントラクトを個別に定義する

### 出力後

設計書のパスをユーザーに伝える:

```
設計書を出力しました: {path}
この設計書に基づいて実装を開始できます。
```

## 禁止事項

- 推測で設計判断を埋めない。不明な点は必ずユーザーに質問する
- コードの変更をしない。この skill の責務は設計書の作成まで
- 要件の不備を見つけた場合はユーザーに報告し、承認を得て要件定義書を直接更新する。design skill のスコープ内で解決できない場合のみ requirements skill での再作成を提案する
- プロジェクトの既存パターンを無視しない。逸脱する場合は理由を明示してユーザーに確認する
- AskUserQuestion を使わずにテキストだけで質問しない
- 設計書にコード実装を書かない。コントラクトの記述に必要な型シグネチャやスキーマ定義は書くが、関数の中身やロジックの実装は書かない
- 成果物に策定過程の記録を含めない。セルフレビューでの指摘・修正、上流文書の不備発見・修正等の作業履歴は成果物に書かない。成果物には最終状態のみを記載する
- 実装 agent (impl 等) の編集境界外にあるファイルを plan の変更対象に含めない。プロジェクトに protected directory (実装 agent から編集を禁じられた領域、例: Claude Code の `.claude/` 配下) がある場合、その編集はユーザー責務として plan から分離する。plan に含めると実装 agent が plan エラーで停止するか、agent が境界を越えて編集することで設計ミスが顕在化しなくなる
