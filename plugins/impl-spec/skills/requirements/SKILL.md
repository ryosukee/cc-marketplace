# requirements

コード実装の前段として、「何を作るか」を要件レベルで明確化する。
出力は要件定義書。設計判断 (どう作るか) はこの skill のスコープ外で、design skill の責務。

## 横断原則: プロジェクト方針への適応

この skill はプロジェクト非依存だが、使われるプロジェクトに fit する。
全フェーズを通じて以下を守る:

- Phase 0 で CLAUDE.md / rules / ドキュメント規約を読み取り、以降の全フェーズで参照する
- 調査はプロジェクトで利用可能なツールを優先する (Serena MCP, LSP 等があれば使う。なければ Read / Grep / Glob / Agent で代替)
- 質問の粒度・観点はプロジェクトのドメインとアーキテクチャに合わせる
- 出力ファイルの配置先・命名・フォーマットはプロジェクトの規約に従う
- プロジェクトの用語を使う。プロジェクト内で確立された概念は skill 側で再定義しない

「特定プロジェクトの仕組みを知っている」のではなく、「目の前のプロジェクトを読んで合わせる」。

## Phase 0: プロジェクト文脈の把握

他のフェーズに先立ち、プロジェクトの方針を理解する。

### 読み取る対象

1. `CLAUDE.md` (概要、技術スタック、ディレクトリ構成、コーディング規約、ワークフロー)
2. `.claude/rules/` (プロジェクト固有のルール。paths 付きルールは対象範囲を確認。最も信頼度の高いソース)
3. プロジェクト内のドキュメント: 設計方針、既存の仕様書・要件書・設計書・ADR 等を Glob で探索する。配置場所はプロジェクトによって異なる (`docs/`, `plans/`, `specs/` 等)
4. 利用可能な MCP サーバー (Serena 等のコード解析ツールがあるか)

Phase 0 で把握した情報は以降の全フェーズで照合基準として使う。Phase 2 の調査、Phase 4 の品質チェックで、ここで把握したルールやドキュメントと矛盾する記述がないか常に検証する。

Phase 0 の結果をユーザーに報告する必要はない。内部的に保持して以降で参照する。

## Phase 1: 入力の理解

引数を解釈する。

- ファイルパスが渡された場合: そのファイルを Read して概要として取り込む
- テキストが渡された場合: そのまま概要として保持する
- 引数なしの場合: AskUserQuestion で「何を実装したいですか?」と聞く

取り込んだ概要から、実装の対象範囲を推定する。
この時点では推定にとどめ、確定は Phase 3 で行う。

## Phase 2: コードベース調査

ユーザーに質問する前に、自分で調べられることは調べる。
「コードを読めばわかること」を質問するのは skill の品質に反する。

### 調査ツールの選択

Phase 0 で把握したプロジェクトの利用可能ツールに応じて調査方法を選ぶ。

- Serena MCP がある場合: シンボル解決・型情報・依存グラフを優先的に活用する
- LSP がある場合: 定義ジャンプ・参照検索を活用する
- 上記がない場合: Agent (Explore) を起動して網羅的に調査する

いずれの場合も Read / Grep / Glob は補助的に使える。

### 調査対象

Phase 1 の概要に基づいて以下を調査する:

1. 対象となるコードの構造 (ディレクトリ、主要ファイル、エントリポイント)
2. 関連する既存実装 (類似機能、関連モジュール、共通パターン)
3. 依存関係 (import graph, 共有リソース、外部ライブラリ)
4. テスト構成 (テストファイルの配置、テストフレームワーク)
5. 既に実装済みの部分 (重複実装の回避)
6. プロジェクト固有の制約 (CLAUDE.md や rules に書かれている制約が対象領域に影響するか)

### 調査の精度

既存コードの振る舞いを記述する際は、推測や要約ではなく実装を読んで正確に記述する。
調査結果の不正確さは、下流の設計判断を誤らせる直接の原因になる。

調査結果の検証は Serena MCP を優先し、なければ grep / Read で行う。特に以下は必ず自分で裏付けを取る:

- 否定的記述 (「存在しない」「していない」「のみ」): 不在を確認する
- 固有名詞 (関数名、ファイルパス、コンポーネント名): 実在を確認する

Agent に委任した調査結果も同様に検証する。Agent は「見つけられなかった」と「存在しない」を区別しない。

Phase 0 で把握した規約・既存ドキュメントを調査の照合基準として使う。

### 調査結果の報告

調査結果をユーザーにテキストで報告する:

- 対象領域の現状 (何がどこにあるか。ファイルパスと行番号を添える)
- 既に実装済みの部分 (あれば)
- 関連する既存パターン
- 質問が必要な未確定事項の予告

## Phase 3: インタビュー

AskUserQuestion を使って、要件の曖昧さを解消する。
設計判断 (どう実装するか) ではなく、要件 (何を作るか) に集中する。

### 質問の原則

- 自明な質問をしない。Phase 2 の調査で判明したことは聞かない
- 上流の判断を先に確認してから下流の詳細を聞く
- AskUserQuestion の選択肢形式を活用する
- 独立した質問は 4 問までまとめて聞く
- 5 問以上ある場合は複数回に分ける。分ける場合は事前に全体計画を提示する

### 質問のカテゴリ (要件レベル)

以下の観点から質問を構成する。Phase 2 で判明済みの観点はスキップ。

1. スコープ: 含むもの / 含まないもの。MVP の境界
2. 機能要件: 正常系の動作仕様。ユーザーから見た振る舞い
3. エッジケース: 境界値、異常入力、同時操作、データなし時
4. エラーハンドリング: エラー時にユーザーに何を見せるか、リカバリ方法
5. 状態遷移: 要件が時間経過で状態が変わる振る舞いを含む場合、各状態とその遷移条件を明確にする
6. 暗黙の技術的前提: 要件の実現が既存の仕組みの変更を前提とする場合、その変更自体をスコープに含めるかを上流判断として確認する
7. ドメイン概念の粒度: 要件中に同種・同時・同一などの集合的な概念が出てきた場合、その境界と分類基準を定義する
8. UI/UX: 表示、操作、フィードバック (UI がある場合)
9. 既存機能との整合性: 影響範囲、breaking change の許容
10. 非機能要件: パフォーマンス、セキュリティ、アクセシビリティ (該当する場合)
11. 受入基準: この機能が「完成した」と言える条件

設計の選択肢は聞かない。
ただしユーザーが要件レベルで技術的制約を表明した場合は要件として記録する。

### インタビューの進め方

1. まずスコープと優先度を確定する (最も上流の判断)
2. スコープ内の各機能について、正常系 → エッジケース → エラーの順で詰める
3. 各ラウンドの質問を送る前に、前の回答を踏まえて次に何を聞くか判断する
4. 必要に応じて追加のコードベース調査を挟む
5. 曖昧さがなくなったと判断したら終了を宣言する

### 終了条件

以下の全てを満たしたらインタビュー終了:

- スコープが明確 (含む / 含まないが決まっている)
- 各機能の正常系の動作仕様に曖昧さがない
- エッジケースとエラーの扱いが決まっている
- 状態遷移が必要な要件では、各状態と遷移条件が定義されている
- 受入基準が定義されている

終了を判断するのは skill 側。「まだ質問はありますか?」とは聞かない。

## Phase 4: 要件定義書の出力

インタビュー結果をまとめた要件定義書を Markdown ファイルとして出力する。
出力前に spec-reviewer agent でセルフレビューを行い、指摘があれば修正してから出力する。

### セルフレビュー

ファイルに書き出す前に、Agent ツールで `spec-reviewer` を起動する。

- subagent_type: `impl-spec:spec-reviewer`
- 入力: 要件定義書のドラフト全文、doc_type は `requirements`
- 出力: 指摘リスト

指摘への対応は種類によって分ける:

- skill が自力で修正できるもの (曖昧表現の書き換え、内部整合性の修正等): そのまま修正する
- ユーザーへの再質問が必要なもの (未確定項目、スコープの暗黙前提、要件の不足等): Phase 3 に戻り AskUserQuestion で確認してから修正する

### 出力先の決定

プロジェクトの規約に従う:

1. 仕様書用のディレクトリがあればそこに配置
2. 既存の仕様書があればその隣に配置
3. どちらもなければ `docs/` に配置

ファイル名はプロジェクト規約に従う。なければ `req-{feature-name}.md` の形式。

### 要件定義書の構成

プロジェクトにテンプレートがあればそれに従う。なければ以下の構成:

1. 概要: 何を実装するか (1-2 文)
2. 背景: なぜ必要か
3. スコープ: 含むもの / 含まないもの
4. 機能要件: 各機能の動作仕様
    - 正常系の振る舞い
    - エッジケースの扱い
    - エラー時の振る舞い
    - 状態遷移 (該当する機能のみ)
5. 非機能要件 (該当する場合)
6. 技術的制約: ユーザーが要件レベルで表明した制約 (あれば)
7. 受入基準: この機能が完成したと言える条件のリスト
8. 未決事項: 保留にしたもの (あれば)

### 要件定義書の品質基準

- 各要件は検証可能な記述にする
- 曖昧な表現 (「適切に」「必要に応じて」「など」) を残さない
- design skill がこの文書だけで設計判断に進めるレベルの情報量を持つ
- 設計方法には踏み込まない。「何を」だけを書き、「どう」は書かない
- 受入基準の各項目に対して、機能要件または技術的制約の中に検証を可能にする定義があることを確認する。「設計判断に委ねる」と書いた項目が受入基準と矛盾しないか検証する
- Phase 0 で把握したルール・ドキュメントと矛盾する記述がないことを確認する

### 出力後

要件定義書のパスをユーザーに伝え、design skill への接続を案内する:

```
要件定義書を出力しました: {path}
設計書の作成に進む場合は /impl-spec:design {path} で開始できます。
```

## 禁止事項

- 推測で要件を埋めない。不明な点は必ずユーザーに質問する
- コードの変更をしない。この skill の責務は要件の明確化まで
- 設計判断に踏み込まない
- プロジェクトの規約を無視しない
- AskUserQuestion を使わずにテキストだけで質問しない
