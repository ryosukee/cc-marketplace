# 入り組んだ説明・報告・確認は HTML で行う

調査報告・設計判断の比較・設問の多い確認など、前提や構造が入り組んだ内容は、
ターミナルへのテキスト出力ではなく self-contained な HTML ページを生成して見せる。
構造化して順序立てて読める形に作り込むこと自体が目的なので、体裁は手を抜かない。

## 適用基準

- HTML にする: 前提説明が長い調査報告、複数の表・比較を含む設計判断、設問が概ね 4 問を超える確認、順序立てた説明が必要なもの
- テキストのまま: 短い報告、単発〜少数の質問（レンダリングバグ修正までは AskUserQuestion を使わず、
  チャットのフリーテキストで質問する。[ユーザーへの確認は選択肢形式で](../ask-with-choices/SKILL.md) 冒頭の警告参照）

## 配置先と URL の解決

配置先ディレクトリと配信 URL は環境変数から解決する（値のセットアップは環境側の文書の管轄）。

- `CLAUDE_PAGES_DIR`: 共通ページディレクトリ。未設定なら既定値 `~/.local/share/claude-pages` を使う
- `CLAUDE_PAGES_BASE_URL`: 配信のベース URL（例: `https://<ホスト名>.<tailnet 名>.ts.net`）。
  ページの serve URL は `{CLAUDE_PAGES_BASE_URL}/{ファイル名}`、一覧のルート URL は `{CLAUDE_PAGES_BASE_URL}/`
- `CLAUDE_PAGES_BASE_URL` が未設定・空の場合は、ページ提示の前にユーザーに URL を確認し、
  そのセッションではその値を使う。あわせて環境変数としての恒久設定を提案する

## 生成・表示方法

- 共通ページディレクトリ（`CLAUDE_PAGES_DIR`。無ければ `mkdir -p` で作る）に
  `YYYY-MM-DD-<プロジェクト>-<内容>-<種別>.html`（すべて kebab-case 英語）で書く
    - ブラウザで自動で開かない（`open` コマンドを実行しない）。`file://` 直開きは
      「一覧に戻る」等の相対リンクが serve 側の一覧と繋がらず不便なため。
      閲覧はユーザーが提示された serve URL（またはファイルパス）から自分で行う
    - ユーザーに見せる閲覧用 HTML（確認フォーム・調査レポートを含む全部）はここに置く。
      scratchpad は単一セッション内で使い切る中間物専用で、閲覧用 HTML は置かない
    - 種別は `form`（要回答）/ `report`（読むだけ）の 2 値。連番シリーズのフォームは `form-NN`（ゼロ埋め 2 桁）
    - サブディレクトリは作らない。グルーピングは index.html 側で表現する
    - 改稿は同名上書きにする（URL と日付プレフィックスは初版のまま維持）。別議題は新しいファイル名で作る
    - 各ページの左下に「一覧に戻る」ボタン（`./` = index へのリンク）を常時固定のオーバーレイで置く。
      固定位置はビューポート左端ではなく本文カラムの左端に揃える
      （例: `left: max(12px, calc((100vw - <カラム幅>px) / 2 + <カラム padding>px))`）。
      下部固定フッターがあるページではフッター左端に置いてよい
    - ページ共通の CSS/JS パターン（ライト/ダーク・下書き自動保存・下書きコピー・競合比較 UI・
      一覧に戻る・回答コピー）は、この skill の `templates/page.html` を雛形として使う
- 同ディレクトリの `index.html` で一覧と状態を管理する（Claude が生成・更新。
  エントリは index 内のインライン JS 配列 `entries`）
    - ページ作成時のエントリは必ず未完了で登録する。状態は form → `awaiting`（回答待ち）、
      report → `unconfirmed`（確認待ち）。提示しただけのページを完了扱いにしない
    - 状態遷移は必ず「未完了 → 完了」を踏む: form は「## HTML フォーム回答」を受領したターンで
      `answered`（回答済み）へ、report はユーザーから内容への確認応答（会話での言及・了解）を受けたターンで
      `confirmed`（確認済み）へ更新する
    - 新規ページを追加した index 更新のついでに掃除する: 同じプロジェクトの完了（`answered` / `confirmed`）
      エントリが 10 件を超えていたら、古い順にファイルと index エントリを削除して直近 10 件だけ残す。
      未完了（`awaiting` / `unconfirmed`）は数に関わらず削除しない。
      ユーザーの「掃除して」には指示された範囲で即時対応する
    - セッションが操作してよいのは、自分のプロジェクトのページと index.html（+ PWA 固定アセット）だけ。
      他プロジェクトのページの改稿・削除はしない
    - index の最下部に運用元を明記する: この skill（cc-marketplace の claude-user-communication plugin、
      `plugins/claude-user-communication/skills/html-communication/`）に基づく Claude Code の
      HTML コミュニケーション用ディレクトリである旨と、skill の GitHub URL を footer に書く。
      index を再生成するときも維持する
    - PWA 固定アセットを配置する: `manifest.json`（name = claude-pages、start_url = index.html、
      display = standalone）と `icon-192.png` / `icon-512.png`。index の `<head>` に
      `<link rel="manifest">` と theme-color を入れる。「閲覧用 HTML のみ」の例外はこれらと index.html だけ
    - index・PWA アセットが消えていたら、この skill の `templates/`
      （index.html 雛形・manifest.json・icon-192.png・icon-512.png）から再生成する
- モバイル閲覧: claude-pages は tailnet 内限定の HTTPS 配信にしてある。
  提示時の報告テキストには、ファイルパス / ページの serve URL / 一覧のルート URL の 3 つを必ず併記する
- 開き直せるように、ファイルパスを本文でも伝える
- CSS/JS はすべてインラインで self-contained にする。外部 CDN・フォント・画像に依存しない
- `prefers-color-scheme` でライト/ダーク両対応にする
- 構成は「結論・概要 → 前提・経緯 → 詳細 →（あれば）設問」の順。セクション番号か目次を付ける
- 大セクションの見出しには「2/5」のように全体数込みの番号を付ける。読者が全体の分量と現在位置を把握できるようにする
- ターン構造と無関係にブラウザで表示されるため、
  [ユーザーへの確認は選択肢形式で](../ask-with-choices/SKILL.md) に記載のレンダリングバグの影響を受けない

## 情報設計（構成・テーブル・羅列）

情報デザイン norm ライブラリ（product-boilerplate repo の `docs/norms/`、出典付き）に従う。
作成時に特に効く要点（詳細・出典はライブラリ側）:

- 冒頭に地図を置く: この文書の目的 / 決めること（設問）一覧 / 読み方。結論・答えは背景より前に出す
- 1 議題 1 セクション。判断セクションは「決めること → 背景 → 判断材料 → 設問」の同型に揃える
- すべての表・グリッド・カード群の直前にリード文を置く: 「これは何の一覧か」+「どこが決め手か」
- テーブルのセルは流し読みできる短い判定値（体言止め・記号・数値）にし、文を書かない。
  根拠・詳細は表直下の補足リストか脚注へ出す。1 セル 1 情報
- 比較表は「列 = 選択肢、行 = 観点」で列は 5 以下。決め手の行はハイライトし、リード文で行名を名指しする
- 出典・長い補足を本文の括弧書きで埋め込まない。出典はチップ等に分離する
- 回答・判断に不要な資料は末尾の参考資料セクションへ隔離し、「読み飛ばしてよい」と明記する
- フォームには入力の下書き自動保存を実装する（localStorage、入力の都度保存・再訪時に復元。端末ローカル）
    - 「下書きをコピー」ボタンを併設する。整形は回答コピーと同じで、ヘッダだけ
      「## HTML フォーム下書き（{テーマ}）」にする（端末間の下書き持ち運び用。受け取り時の扱いは後述）
    - 「入力をリセット」ボタンを併設する（localStorage の下書きを消して、HTML に埋め込まれた
      デフォルト値へ戻す）
    - 埋め込み下書きと端末の localStorage 下書きの両方が存在して食い違う場合は、
      両方の整形テキストを並べて比較し「どちらを使うか」を選べる UI をページ上部に表示する
      （選ぶまでフォームには埋め込み側を表示する）
- 推奨バッジには根拠の併記と覆し可能性（非推奨側も選べることの明示）を必須にする

## 確認（設問）を含めるときの作り

- 設問は関連するセクションの直下に置く。ページ末尾にまとめて並べない
    - 設問が多いとき、末尾集約は判断材料と設問の往復スクロールが発生して回答しづらい
    - 回答コピー機能は全セクションの設問を集約して 1 テキストにする（配置と集約は別）
- 最終確認などの自由記入欄は、ページ末尾ではなくスクロール追従の下部固定領域に置く（コピー用フッター内など）。
  上から順に読みながら、気づいた点を都度記入できるようにする
- 各設問に選択肢 (radio / checkbox) を用意し、推奨案には推奨バッジを付ける
- 技術選定などの比較設問には、選択肢の説明文だけで判断させない。設問の直前に多観点の比較表
  (機能・性能・実績・エコシステム・リスク・移行性・保守状況など、その選定で効く観点) を置き、
  各観点は調査の裏取り付きで書く。選択肢の description は比較表の結論の要約に徹する
- 選択肢に収まらない回答用に、設問ごとに「その他」の自由記述欄を付ける
- 設問ごとに補足記入欄（任意）を付ける
- ページ末尾に回答の live preview と「回答をコピー」ボタンを置く。
  ボタンはそれまでの選択・自由記述を 1 つのテキストに整形してクリップボードへコピーする
- 整形フォーマットは Claude Code にそのまま貼れる形にする:

  ```text
  ## HTML フォーム回答（{テーマ}）
  - Q1（{設問}）: {回答}
  - Q2（{設問}）: {回答}
  - 補足: {自由記述}
  ```

- クリップボード API が使えない環境向けに、preview の textarea を全選択コピーできるフォールバックを付ける
- 未回答の設問がある状態でもコピーは許可し、未回答は `未回答` と明記する

## フォームの op-review

フォーム類を提示する前に、facet を分担した並列レビューにかけ、
全 facet が approve（指摘 0 件）になるまで修正を繰り返す。
facet の分担・蓄積 norm は [facet 分担と蓄積 norm](./references/facet-review-norms.md) に従う。

## 回答の受け取り

ユーザーから「## HTML フォーム回答」で始まるテキストが貼られたら HTML フォームの回答として扱う。

「## HTML フォーム下書き」で始まるテキストは回答ではなく、端末間の下書き持ち運びとして扱う。
該当フォームの HTML に、下書きの選択・記述を入力のデフォルト値（radio の checked 属性・textarea の初期文字列）
として埋め込んで同名上書きし、ファイルパスと serve URL を再提示する。index の状態は変更しない。
埋め込み時はページ上部に「下書き埋め込み済み（日時）」の注意書きを追加し、以後の埋め込みで日時を更新する。
他端末で開くと下書きが入った状態になり、端末側に別の下書きが残っている場合は比較 UI（前述）で選択できる。
以降の解釈まとめ・作業フローは [ユーザーへの確認は選択肢形式で](../ask-with-choices/SKILL.md) の
「複数回の質問と回答後のフロー」と同じ。
