# session-retrospective

セッション末尾の振り返り。2 フェーズで進める。

## フェーズ 1: codify (rules/skills/CLAUDE.md の更新 → コミット)

ゴール: セッションで得た学び・陳腐化修正・memory 棚卸しを
rules/skills/agents/CLAUDE.md/README に反映し、1 コミットにまとめる。

### 1-1. 棚卸し

以下の 3 つのソースから更新候補を洗い出す。
全てフェーズ 1 のゴール (codify + コミット) に向けた入力。

#### セッション内のやり取り

時系列ではなくテーマごとに整理する。

- 何が繰り返し起きたか
- 何が実害になったか (バグ、差し戻し、やり直し、事故)
- どんな判断ルールが発生したか
- どんなワークフローが機能した / しなかったか

#### 既存ドキュメントの陳腐化

`.claude/rules/`, `.claude/skills/`, `.claude/agents/`,
`CLAUDE.md`, `README.md` をざっと確認し:

- セッション中に使った rules/skills が実態と乖離していないか
- ユーザーの指摘・訂正のうちルール化されていないものはないか
- 使われていないセクション・機能はないか (撤去 or 簡素化)

#### auto memory の棚卸し

memory に蓄積された知見のうち、rule や skill に昇格すべきものが
ないかチェックする。memory は揮発性の一時メモ。
持続的に参照すべき知見は rule/skill に昇格させる。

#### やり残しと次アクションの洗い出し (先行実施)

フェーズ 2 で提示する内容だが、棚卸しと一緒に洗い出しておく。
codify 候補の発見にも役立つため。

やり残し:

- やろうとしていたが着手しなかったこと
- ユーザーから要求があったが完了しなかったこと
- 議論の中で出た改善案で実行されなかったもの
- 途中で方針変更して取りやめたが元の案にも価値があるもの

次アクション提案:

- 実行したタスクへの改善案
- 今回の作業から派生する自然な次ステップ
- 別セッションで取り組むべき課題

### 1-2. 優先度分類

棚卸しの結果から更新候補を抽出し優先度を付ける。

| 優先度 | 基準 | 対応 |
| --- | --- | --- |
| A | 実害があった + 将来再現する | 強く推奨 |
| B | 再利用価値が高い (再現の予兆あり) | 推奨 |
| C | あると便利、実害は小 | 任意 |
| D | 無関係 / 一度きり / 偶発 | 不要 |

各候補は以下の構造で記述する:

```markdown
#### {候補名} [優先度]

- 発見: 何が起きたか / 何がわかったか
- 根拠: なぜそう言えるか (実害、再現条件、具体例)
- アクション: どこに何を書くか (追記先 or 新規作成先)
```

#### 明文化する基準

- **パターンとして再現するか** で判断する (発生回数ではない)
- 1 回でもパターン化できるものは対象
- 実害があったものは 1 回でも対象
- 同じ判断を 2 回以上したものは対象

#### 明文化しない基準

- 偶然性が高い。根本原因が分かってから codify
- 自明すぎるもの
- このセッションだけの特殊状況
- 個人の好みで一般化できないもの

### 1-3. 昇格先の判断

| 知見の性質 | 昇格先 |
| --- | --- |
| `paths:` で対象を絞れる制約 | `.claude/rules/{topic}.md` |
| 特定タスクでのみ参照される知識 | `.claude/skills/ref-{topic}/` |
| 繰り返し実行した対話的な手順 | `.claude/skills/op-{name}/` |
| プロジェクト概要・ドメイン知識 | `CLAUDE.md` |

- 既存ファイルに追記で済むなら追記。新規ファイル乱立を避ける
- 独立したトピックなら新規作成

既存ルールの文言強化は対処療法。
root cause (判断基準の不足、ワークフローの欠陥) が残れば再発する。
「強化で足りるか / 構造変更が必要か」を併せて考える。

### 1-4. ユーザーに提示

以下のフォーマットでレポートを一括提示する。末尾に「対処項目サマリ」
をつけて、これから 1 項目ずつ確認する対象の全体像を見せる。

```markdown
# セッション振り返り

## 発生した出来事 (テーマ別)
1. {テーマ}: 何が起きたか、実害、関連ファイル

## 更新候補

### A (強く推奨)
#### {候補名}
- 発見: ...
- 根拠: ...
- アクション: ...

### B (推奨)
...

### C (任意)
...

## 陳腐化の指摘
- {ファイル}: {何が古いか、どう直すか}

## 対処項目サマリ (全 X 件)
1. [A] {候補名A1}: {一言要約}
2. [A] {候補名A2}: {一言要約}
3. [B] {候補名B1}: {一言要約}
4. [陳腐化] {ファイル}: {一言要約}
...

これから 1 項目ずつ確認します。
```

### 1-5. 対処項目を TaskCreate に登録

レポートの「対処項目サマリ」に載せた全 X 件を TaskCreate で tasks に登録する。
各 task のタイトルはサマリの 1 行と揃えて `[A] {候補名}` 形式にする
(陳腐化なら `[陳腐化] {ファイル}`)。

対話に入る前に tasks を整えることで、途中で中断・脇道に逸れても
tasks から現在位置を復帰でき、どの項目をどう処理したかの履歴も残る。

### 1-6. 1 項目ずつ対話・反映

tasks に登録した各項目について、順に対話・承認・その場で反映を行う。

#### 対話方式

ユーザーへの確認は AskUserQuestion ではなく、その場で 1 項目ずつ
対話しながら決める。候補項目ごとに「この項目どうしますか？」と
問いかけ、ユーザーが議論・方針変更・skip などを自由に返せるようにする。

AskUserQuestion は選択肢が固定化された判断を素早く取るのに向くが、
振り返りの項目は「やる / やらない」の二択で済まないことが多い
(内容調整、別案への差し替え、他項目との統合、skip 理由の議論など)。
対話形式の方がこれらを自然に扱える。

#### 進捗表示

各項目の対話開始時に必ず `[x/n] {項目名}` 形式で進捗を示す。
`x` は現在の項目番号、`n` は tasks の総件数。

例:

```text
[1/5] A: {候補名A1}
発見: ...
根拠: ...
アクション: {追記先 or 新規作成先}

この内容で反映していいですか？
```

- 途中で項目を追加・削除した場合も分母を更新して表示する
- ユーザーが割り込みで別タスクを差し込んだ場合、復帰時に
  「[3/5] に戻ります」と現在位置を明示する

#### tasks の更新

- 対話開始時に対象 task を in_progress
- ユーザーの承認・反映が完了したら completed
- skip した項目も completed 扱い (備考欄に skip 理由を書く)

#### 反映時の注意

承認された項目に関連する変更を **全て** 実行する。
レポートに書いた範囲 (陳腐化修正、昇格、追記) を漏れなく反映する。
一部だけ修正して完了扱いにしない。

- ユーザーが承認した項目のみ反映する。勝手に書かない
- 対象ファイルに既に同じ内容が書かれている場合は重複追記しない

### 1-7. コミット

全項目の対話・反映が完了したら、コミットするかをユーザーに確認する。
コミットする場合は全変更を 1 コミットにまとめる。

## フェーズ 2: やり残しと次アクション

フェーズ 1 のコミットが完了してから提示する。
内容は 1-1 で先行して洗い出し済み。

```markdown
## やり残し
- {未完了項目}: {何をやろうとしていたか}

## 次アクション提案
- {提案}: {なぜやるといいか}
```

やり残しはリストアップのみ。
棚卸し (TODO 化、チケット化等) はしない。
ユーザーが見て判断する材料を提示する。

次アクション提案は押し付けではなく提案。採用はユーザー任意。

## 引き継ぎとの連携

context が上限に近く次セッションに作業を引き継ぐ必要がある場合は、
本 skill 完了後に `/session-closing:handover` を呼んで
`HANDOVER.md` を project root に書き出す。

本 skill は「セッションで得た学びの codify」、handover は
「次セッションへの状態引き継ぎ」で責務が違う。両方必要な session では
retrospective → handover の順で呼ぶのが自然。

## 注意事項

- 勝手に書き換えない。ユーザー承認を必ず経る
- 過剰に抽出しない。「とりあえず codify」はノイズ化の原因
- 承認された項目は関連する変更を漏れなく反映する
