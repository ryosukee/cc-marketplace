# ワークフロー診断 & 構成

対象プロジェクトの `.claude/` をスキャンし、ユーザーが登録した参考リポジトリと本 skill の原則に基づいて、状態に応じた修復・更新・セットアップを行う。

## 意図

初回セットアップも既存プロジェクトの改善も同じ skill で扱う。プロジェクトに「何があるか」「何が足りないか」「何がズレているか」を診断した上で、ユーザーがモードを選び、参考リポジトリから具体例を抽出しつつ原則に沿って合成する。

固定テンプレートは持たない。参考リポジトリは実プロジェクトの `.claude/` を参照するのみで、都度 fetch する。この skill 自体が「何を抽出し、どう組み合わせるか」の方針を保持する。

## 前提

- 参考リポジトリの一覧: `${CLAUDE_PLUGIN_DATA}/registry.json`
- 加えて、ユーザーグローバル dotclaude root (`$HOME/.claude`) を **暗黙エントリ** として常に対象に含める。registry.json への登録は不要で、どの環境にも存在する前提
- 一覧が空またはファイル不在でも、暗黙エントリ (`$HOME/.claude`) がワークフロー構成として有効なら処理を続行する。暗黙エントリも無効で registry も空なら `/dotclaude:registry add {owner/repo}` での登録を促して終了する

### 暗黙エントリ (`$HOME/.claude`) の扱い

registry の通常エントリと同列に扱うが、以下の点だけ異なる:

| 項目 | 値 |
|---|---|
| name | `user-global` |
| github | なし (ローカル専用) |
| role | `primary` |
| owned | `true` |
| note | ユーザーグローバルの dotclaude root。全プロジェクト共通のルール・skill・agent が置かれる |
| base dir | `$HOME/.claude` (subpath 解決なし) |

有効性チェック (`.claude/agents/` または `.claude/skills/` に 1 ファイル以上) は通常エントリと同じ。`$HOME/.claude` 自体が `.claude` 相当なので、`$HOME/.claude/agents/` と `$HOME/.claude/skills/` を直接見る。

## ワークフロー

### ステップ 1: 前提チェック

1. `${CLAUDE_PLUGIN_DATA}/registry.json` を読む
2. 暗黙エントリ (`$HOME/.claude`) の有効性を確認する (`$HOME/.claude/agents/` または `$HOME/.claude/skills/` に 1 ファイル以上あるか)
3. registry が空/不在 **かつ** 暗黙エントリも無効 → 以下のメッセージを表示して終了:
   ```
   参考リポジトリが登録されていません。
   先に /dotclaude:registry add {owner/repo} で 1 つ以上登録してください。
   ```
4. cwd のプロジェクト名と git remote を取得する (`git remote get-url origin` 等)

### ステップ 2: 自己除外

現在のプロジェクトが registry の参考リポジトリに含まれる場合、それを fetch 対象から除外する:

1. git remote から owner/repo 形式の識別子を取得、現在の cwd から repo root への相対パスも取得
2. registry の各エントリの `github` フィールドと照合 (subpath 付きエントリは先頭 2 セグメントが owner/repo と一致し、かつ subpath が現在の cwd 相対パスと一致する場合のみ自リポジトリ扱い)
3. cwd が `$HOME/.claude` 配下の場合、暗黙エントリ (`user-global`) も除外リストに入れる
4. 一致したものを除外リストに入れ、「{name} は現在のプロジェクト自身なので参考リポジトリから除外しました」と報告

残った参考リポジトリが 0 件になった場合は、比較対象がないため登録を促して終了する。

### ステップ 3: 参考リポジトリの fetch

registry の各エントリに加え、暗黙エントリ (`$HOME/.claude`) を常にリストに含めて処理する (ステップ 2 で除外された場合を除く)。暗黙エントリは `github` フィールドがなく、base dir は `$HOME/.claude` 固定、subpath 解決・ghq 検索・gh api fetch をスキップして直接読む。

各参考リポジトリについて以下を試みる:

1. registry の該当エントリから `note` も合わせて読み込む (合成時のヒントとして使う)
2. `github` フィールドをパース: 最初の 2 セグメントを `{owner}/{repo}`、残りを `{subpath}` として扱う (例: `omnisinc/efso-document/to-be/idp` → owner/repo=`omnisinc/efso-document`, subpath=`to-be/idp`)
3. `ghq list --full-path` で `{owner}/{repo}` に一致するローカルクローンを探し、あれば `{local}/{subpath}` を base dir として使う (subpath なしならクローン root)
4. 見つかればそこから `.claude/` を読む
5. 見つからなければ `gh api repos/{owner}/{repo}/contents/{subpath}/.claude` (subpath なしなら `/.claude`) で取得
6. 失敗したらそのリポジトリをスキップし、理由を記録する

#### 有効性チェック

取得できたリポジトリについて、ワークフロー構成として有効か確認する。`.claude/agents/` もしくは `.claude/skills/` のいずれかに 1 ファイル以上あれば有効。どちらも空ならスキップし、理由を記録する。

全リポジトリがスキップされた場合は、スキップ理由一覧を表示して終了する。

#### note の扱い (重要)

registry の `note` は「このリポジトリのどの部分を参考にし、どの部分を参考にしないか」のヒントとして使う。例:

- 「パイプライン部分は runner 実装に依存しているので抽出しない。個別 skill のみ参考」
- 「agents は成熟しているが rules は初期段階で未整備」

note が書かれているリポジトリでは、doctor は合成時にその指示を尊重する (note で「無視」と書かれた部分は抽出対象から外す)。

ただし note を盲信せず、実際に repo を読んだ結果と突き合わせる。note と現実に矛盾があれば、ステップ 5 のレポートで報告する (「note には X と書かれているが、現実には Y が存在する。note が古い可能性がある」)。

### ステップ 4: プロジェクト診断

現在の `.claude/` と `CLAUDE.md` をスキャンし、状態を把握する:

- `.claude/agents/*.md` の一覧と内容
- `.claude/skills/*/` (`SKILL.md` または `skill.md`) の一覧と内容
- `.claude/rules/*.md` の一覧と内容
- `CLAUDE.md` の有無と内容
- `.claude/settings.local.json` の有無

参考リポジトリと照合して、以下を分類する:

| 分類 | 内容 |
|---|---|
| 一致 | 参考リポジトリと同等のファイルが存在 |
| 不足 | 参考リポジトリにあるがプロジェクトにない役割のファイル |
| 既存・差分あり | 両方にあるが構造・手順・セクションが異なる |
| プロジェクト独自 | 参考リポジトリにないがプロジェクトにある。価値の有無は内容で判断 |

状態の全体判定を出す:

- **空**: ワークフロー系ファイルがほぼない
- **部分的**: 一部だけ揃っている
- **完備**: 役割が網羅されている
- **ドリフト**: ファイルはあるが参考リポジトリと方向性が大きく違う

### ステップ 5: 診断レポートとモード選択

診断結果をレポートとして表示する:

```markdown
# dotclaude Doctor Report

## プロジェクト: {project_name}
状態: {空 / 部分的 / 完備 / ドリフト}

## 参考リポジトリ
- {repo1} (primary) — note: {note 要約}
- {repo2} (reference) — note: {note 要約}
- (除外) {self_repo}: 自己

## note と現実の不一致 (あれば)
- {repo1}: note には「X」と書かれているが、実際には {Y} が存在する。note が古い可能性がある
  - 対応候補: このまま進む / note を更新する / 該当部分を無視する

## 現状サマリー
- agents: N 個 ({list})
- skills: N 個
- rules: N 個
- CLAUDE.md: あり/なし

## 分類
### 一致
{list}

### 不足
{list}

### 差分あり
{file}: {差分の要約}

### プロジェクト独自
{file}: {内容の要約、価値の推定}
```

その後、AskUserQuestion で実行モードを選ばせる:

| モード | 動作 |
|---|---|
| 差分アップデート | 既存の方針・構造は尊重。不足分の追加 + 既存ファイルへの部分的な追記・修正はあり得る |
| エッセンス保持再構成 | 参考リポジトリの方針に寄せて大幅書き換え。既存の価値あるエッセンスは救出して統合 |
| リセット & 再生成 | 既存のワークフロー系ファイルを削除し、参考リポジトリと原則からゼロ合成 |
| レポートのみ | 変更しない |

### ステップ 6: ユーザー文脈の確認

プロジェクトの tech stack・方針を確認する。`CLAUDE.md` や既存ファイルから推定できる項目は推定値を提示し、確認・修正してもらう。推定できない場合は AskUserQuestion で質問する。

確認・質問する主な項目:
- メイン言語
- フロントエンドフレームワーク (ある場合)
- ビルドコマンド
- lint コマンド
- テストフレームワーク / テストコマンド
- E2E フレームワーク (ある場合)
- UI 変更時のスクリーンショット確認方針
- アーキテクチャパターン
- テスト方針 (ユニット重視 / E2E 重視 / バランス等)

質問は AskUserQuestion で 4 問ずつに分ける。既に明らかな項目はスキップする。

### ステップ 7: 合成計画の提示

選択されたモードとユーザー文脈を踏まえて、これから何を生成・更新・削除するかの計画をレポートとして出す。各項目に対し、ユーザーが個別に承認/スキップできる形にする。

```markdown
## 実行計画

### 生成 (新規ファイル)
- .claude/agents/{name}.md — {役割の説明}
  - 参考: {ref_repo}/{path}
  - 原則: {適用する原則}

### 更新 (既存ファイル)
- .claude/agents/{name}.md — {変更内容}
  - 既存の保持部分: {内容}
  - 変更部分: {diff}

### 削除 (リセットモードのみ)
- .claude/agents/{name}.md

### 現状維持
- .claude/agents/{name}.md (プロジェクト独自、価値あり)
```

AskUserQuestion で「全て承認」「個別に確認」「キャンセル」を選ばせる。

### ステップ 8: 合成生成

承認された項目について、以下の順序で生成する:

1. `.claude/agents/` の生成・更新
2. `.claude/skills/` の生成・更新
3. `.claude/rules/` の生成・更新
4. `CLAUDE.md` の生成・更新
5. `.claude/settings.local.json` の提案 (既存がなければ)

各ファイルの合成は本 skill の「合成原則」セクションに従う。

### ステップ 9: 完了報告

生成・更新・削除したファイル一覧と、次のステップを案内する。

### ステップ 10: note の更新 (任意)

ステップ 5 で note と現実の不一致を報告した参考リポジトリについて、ユーザーに note を更新するか確認する。

1. 不一致があったリポジトリを列挙
2. 各リポジトリについて AskUserQuestion で「note を更新する / そのまま / 後で review する」を選ばせる
3. 「更新する」を選ばれた場合:
   - 現在の note と、不一致内容を踏まえた更新案を提示 (ユーザーが自由に修正可能)
   - 確認後、`${CLAUDE_PLUGIN_DATA}/registry.json` を読み込み、該当エントリの note を上書きして保存
4. 更新したエントリ一覧を報告

このステップは合成処理そのものとは独立しており、合成をスキップしても単独で実行できる。レポートのみモードでも note の不一致があれば更新確認を行う。

## 合成原則

この skill がテンプレートを持たない代わりに encode している原則。合成時はこれらを念頭に、参考リポジトリから具体例を抽出して組み合わせる。

### 抽出の基本姿勢

参考リポジトリの `.claude/` 配下にある agent/skill/rule は **全てスキャン対象** とする。実装パイプラインに関わるものだけでなく、ドキュメント整備、調査、レビュー、リファクタリング支援、運用ツール、メタ作業 (rules/agents の authoring 支援) など、およそ有用そうなものは広く提案候補に入れる。

対象プロジェクトの特性を踏まえて、どれを取り込むべきか判断する。プロジェクトの特性を把握するために以下を確認する:

- **プロジェクトタイプ**: library / application / CLI tool / documentation site / research notes / monorepo 等
- **開発フェーズ**: 初期 / 成長期 / 成熟期 / メンテナンス
- **チーム規模**: ソロ / 小規模 / 大規模
- **主要活動**: コード実装中心 / ドキュメント中心 / 調査中心 / 混在
- **既存のワークフロー成熟度**: `.claude/` の現状と CLAUDE.md の内容から推定

特性に応じて取捨選択する。例:

- ソロの research プロジェクトにチーム開発向けパイプラインを丸ごと持ち込む必要はない (過剰)
- ドキュメント中心の repo にはコード実装 agent より markdown レビュー skill の方が有用
- 成熟期のプロジェクトには authoring 系 rules (rule-authoring, claudemd-authoring 等) が刺さる
- 初期プロジェクトには重たいパイプラインより軽量な plan → implement の 2 段階が合う

### 役割カテゴリ例 (網羅ではなく例示)

参考リポジトリに見つかる可能性のある役割の例。**これは抽出対象の限定ではなく、パターン認識の手がかり**。参考リポジトリにこれ以外の役割があれば、それも対象プロジェクトに有用か判断して提案する。

#### 実装パイプライン系

- 実装を行う役割 (plan ベース)
- テスト設計を行う役割 (コードは書かない)
- テスト実装を行う役割
- コードレビューを行う役割 (書き込み権限なし、差し戻し)
- テストレビューを行う役割
- テスト実行を行う役割 (軽量モデルで自動報告)
- メタレビュー (ルール・プラン・定義とコードの整合性チェック)
- 対話的プラン作成
- パイプラインオーケストレーション (複数役割の順次・並列起動、差し戻しループ管理)

#### ドキュメント・文章系

- markdown 文体レビュー / humanize
- README・CLAUDE.md の authoring 支援
- system documentation の作成・レビュー
- 議事録・変更履歴の整理

#### 調査・探索系

- コードベース調査 / 実態把握
- 依存関係・影響範囲の分析
- 外部ライブラリ・ドキュメント調査

#### メタ・authoring 系

- rules ファイルの authoring (frontmatter 規約、1 トピック 1 ファイル等)
- CLAUDE.md の authoring (サイズ制限、役割分担)
- agent/skill 定義の authoring
- パーミッション設定の棚卸し
- プラグイン・ツール設定のセットアップ

#### 運用・自動化系

- GitHub Actions ログ確認、PR 作成、issue 調査
- commit メッセージ生成 / コミット分割
- スケジュール実行、定期チェック
- 外部 API 連携、通知

#### リファクタリング・品質系

- コード簡素化 (simplify)
- dead code 検出
- 命名統一、重複検出

これらに加え、参考リポジトリに独自の役割があれば、その意図を汲み取って対象プロジェクトへの有用性を判断する。

### パイプライン原則

参考リポジトリから抽出したパイプラインを構成する際、以下の原則を守る:

- **責務分離**: 実装・設計・レビュー・実行は別の役割として分ける
- **レビューは read-only**: レビュー役割は書き込み権限を持たず、差し戻しで再実装を依頼する
- **並列実行は read-only のみ**: 複数の役割を並列で動かす場合、ファイル編集の競合を避けるため書き込み権限のあるものは並列化しない
- **差し戻しループに上限**: 無限ループを避けるため、差し戻しの最大回数を定義する (例: 最大 3 回)
- **オーケストレーターはコードを書かない**: パイプラインを制御する側は、自分で実装やレビューを行わず、役割の起動と結果確認に専念する
- **テスト失敗時は実装に戻す**: テスト実行で失敗したら実装の役割に差し戻して修正させる (こちらも上限あり)
- **メタレビューは最後**: 実装・テスト・レビューが完了した後に、ルールやプランが実態とズレていないかを検証する

### ルールファイルの authoring 原則

`.claude/rules/*.md` を生成する際の原則:

- **1 トピック 1 ファイル**: ファイル名はケバブケースで内容を表す
- **frontmatter の `paths`**: YAML frontmatter で glob パターンの `paths` を使い、適用対象ファイルを制限する。`paths` がないルールはセッション開始時に無条件で読み込まれる
- **具体的に書く**: 「コードを整理する」ではなく「handler 層に SQL を書かない」のように検証可能な指示にする
- **CLAUDE.md と重複しない**: CLAUDE.md は概要 + リンク、rules に詳細を書く
- **ルール間で矛盾しない**: 矛盾すると Claude が任意に選択する
- **コードサンプルは実コードと一致させる**: 陳腐化したサンプルは誤ったコード生成の原因になる

### CLAUDE.md authoring 原則

プロジェクトの `CLAUDE.md` を生成・更新する際の原則:

- **役割**: セッション開始時に読み込まれるプロジェクトガイド。コードの書き方ではなく、構造・ビルド方法・ドメイン知識を書く
- **サイズ**: 200 行以下を目標とする。超えそうなら rules ファイルに分離するか `@path` で import する
- **書くべきもの**: ビルド・起動・検証コマンド、ディレクトリ構成、ドメインモデル (ER 関係) と主要フロー、環境構成、既知の制約、ワークフロー規約
- **書かないもの**: 言語・フレームワーク固有のコーディングルール、テスト規約の詳細、コンポーネント設計の詳細。これらは rules ファイルに分離する
- **重複禁止**: rules に詳細があれば CLAUDE.md は概要 + リンクのみ
- **ドメインモデル更新**: 機能追加・DB スキーマ変更・フロー変更時に「ドメインモデル」「主要フロー」セクションを更新する

### agent / skill authoring 原則

`.claude/agents/` と `.claude/skills/` を生成する際の原則:

- **冒頭に frontmatter**: `name`, `description`, `allowedTools` (agent の場合) を書く
- **意図セクション**: そのファイルが「何のためにあるか」「なぜそう設計されているか」を短く書く。後から読む Claude が役割を理解できるようにする
- **手順は箇条書きで明確に**: 実行ステップを順序立てて書く
- **ルール・注意事項**: 守るべき制約・やってはいけないことを別セクションで明示する
- **出力フォーマット**: agent の場合、報告形式を明記する
- **read-only 役割は `allowedTools` から `Write`/`Edit` を外す**: レビュー役割等

### 合成時の手順

1. 参考リポジトリの `.claude/` を読み込む
2. 各ファイルを役割カテゴリに分類する
3. プロジェクトに不足している役割、または差分のある役割を特定する
4. 選択されたモードに応じて処理:
   - **差分アップデート**: 不足分を生成し、既存ファイルに原則に基づく部分的な追記・修正を提案
   - **エッセンス保持再構成**: 既存ファイルの価値ある部分 (プロジェクト固有の制約、独自ルール、有用なカスタマイズ) を抽出し、参考リポジトリ構造をベースに再合成
   - **リセット**: 既存を削除し、参考リポジトリ + 原則 + ユーザー文脈から新規合成
5. 生成する各ファイルは、参考リポジトリのセクション構成を踏襲しつつ、tech stack 固有の記述はユーザー文脈から埋める
6. 参考リポジトリが複数ある場合、**単純に 1 つの repo を選んでコピーしない**。同じ役割を担うファイル群を全参考リポジトリから集めて差分を取り、各 repo の良い部分をマージした合成版を生成する。cross-review skill の「役割クラスタリング + マージ合成」と同じアプローチを取る:
    - 役割を意味的にクラスタリング (名前違いでも同役割なら同クラスタ)
    - クラスタ内で差分分析: マージ可能 (相補的) / マージすべきでない (tech stack 固有の個性) / 古い方が不要 / 競合の 4 分類
    - マージ可能な差分は統合。個性部分は除外 (ただし対象プロジェクトと同じ tech stack を使う repo の固有記述は流用候補)
    - 判断に迷う差分はユーザーに相談する。コンテキストを省略せず十分提示する
7. 合成版はあくまで「参考リポジトリから生まれた最新版」であり、対象プロジェクトの tech stack・特性に合わせて最終調整してから配置する

### 合成時の注意事項

- **参考リポジトリの内容を丸コピーしない**: 具体的なビルドコマンドやファイルパスはプロジェクトに合わせて書き換える
- **プロジェクト独自の価値を失わない**: エッセンス保持モードでは、参考リポジトリにない独自ファイルやカスタマイズを意図的に保持する
- **tech stack 非依存部分は共通化**: パイプライン制御・レビュー観点・BDD 原則など、tech stack に依存しない記述は参考リポジトリからそのまま踏襲して良い
- **tech stack 固有部分は合成**: ビルドコマンド・テストコマンド・フレームワーク固有のパターンはユーザー文脈から生成する
- **不明な点はユーザーに確認**: 推定で進めて誤った合成をするよりも、AskUserQuestion で確認する
