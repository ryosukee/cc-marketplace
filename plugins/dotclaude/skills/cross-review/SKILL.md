# リポジトリ横断レビュー

`${CLAUDE_PLUGIN_DATA}/registry.json` に登録された全参考リポジトリを fetch し、相互に比較して、`owned: true` のリポジトリに対する改善提案を出す。

## 意図

doctor は「今いるプロジェクト」を対象にするが、cross-review は「registry 全体の相互学習」を担う。複数リポジトリでワークフローを育てていると、A repo で発展した手順が B repo にまだない、といったズレが生じる。cross-review はそれを検出し、自分の持ち物 (owned) に反映する提案を出す。

他人のリポジトリに対しては改善提案を出さない (PR を出せないため無意味)。owned=true のリポジトリに対してのみ「この部分は別 repo から取り込めそう」と提案する。

## 前提

- registry: `${CLAUDE_PLUGIN_DATA}/registry.json`
- 登録リポジトリが 2 件以上必要 (1 件だけなら比較対象がない)
- `owned: true` のリポジトリが 1 件以上必要 (出力先がない)

## ワークフロー

### ステップ 1: 前提チェック

1. `${CLAUDE_PLUGIN_DATA}/registry.json` を読む
2. リポジトリが 2 件未満 → 「比較には 2 件以上の参考リポジトリが必要です」と表示して終了
3. `owned: true` のエントリが 0 件 → 「改善提案の出力先となる自分のリポジトリが登録されていません。`/dotclaude:registry add` で登録する際に owned を true に設定してください」と表示して終了

### ステップ 2: 参考リポジトリの fetch

各エントリについて以下を試みる:

1. registry から `note` も合わせて読む (ヒントとして使う)
2. `ghq list --full-path` でローカルクローンを探す
3. 見つかればローカルから `.claude/` を読む
4. 見つからなければ `gh api repos/{owner}/{repo}/contents/.claude` で取得
5. 失敗したらスキップし、理由を記録する

#### 有効性チェック

`.claude/agents/` または `.claude/skills/` に 1 ファイル以上あれば有効。どちらも空ならスキップ。

### ステップ 3: 横断分析

取得した全リポジトリの `.claude/` を相互に比較する。

#### スキャン対象

各 repo の `.claude/` 配下にある agent/skill/rule は **全てスキャン対象** とする。実装パイプライン関連 (impl, test-plan, review 等) だけでなく、ドキュメント・文章系、調査・探索系、メタ/authoring 系、運用・自動化系、リファクタリング・品質系など、参考になる可能性のあるものは広く拾う。

#### 役割の識別

各ファイルが何の役割を担っているかは以下から推定する:
- ファイル名のヒント
- frontmatter の `description`
- ファイル内容の「意図」セクション、手順、ルール

例えば「実装」「テスト計画」「コードレビュー」「メタレビュー」「markdown humanize」「パーミッション棚卸し」「調査 agent」「GitHub Actions 確認」など、固有のカテゴリに囚われず repo にあるまま認識する。

#### ギャップ検出

owned=true の repo について、以下のギャップを検出する:

1. **役割の欠損**: 他の repo にあってこの repo にない役割。実装パイプラインに限らず、ドキュメントレビュー、調査、メタ作業、運用自動化などあらゆる種類の agent/skill/rule が対象
2. **手順・セクションの欠損**: 同じ役割のファイルが両方にあるが、片方にしかない手順・セクション・ルールがある
3. **パイプライン構造のズレ**: オーケストレーション skill の実行フロー (フェーズ分割、並列実行、差し戻しループ上限等) が他の repo と異なる
4. **rules の authoring 品質**: `paths` frontmatter の書き方、1 トピック 1 ファイルの原則、CLAUDE.md との重複状況等
5. **汎用 skill/rule の有用性**: 他の repo にある汎用的な skill (例: markdown humanize、simplify、調査 agent 等) が、この repo にも導入する価値があるか

#### repo 特性の考慮

各 owned repo の特性 (プロジェクトタイプ、開発フェーズ、主要活動、チーム規模等) を推定し、提案の取捨選択に使う。例:

- ドキュメント中心の repo には markdown 系 skill を優先して提案
- コード実装中心の repo には実装パイプライン系を優先
- ソロ運用の research repo には重たいチーム向けパイプラインは提案しない
- 既に成熟している領域には変更を提案しない (過剰な変更を避ける)

#### note の考慮

各 repo の `note` を読んで、「この部分は参考にしない」と記されている領域は分析対象から外す。

例: kanban-agent の note に「パイプラインは runner 依存」とあれば、kanban-agent から owned repo にパイプライン skill を移植する提案は出さない。逆に、owned repo の note に「パイプラインは独自実装」とあれば、他 repo のパイプラインを参考にする提案は出さない。

ただし doctor と同様、note を盲信せず現実と突き合わせる。矛盾があればステップ 5 で報告する。

### ステップ 4: 提案の生成

owned=true の各 repo について、以下の形式で提案をまとめる:

```markdown
# Cross Review Report

## 対象: {owned_repo_1} (owner/repo)

### ギャップ 1: {概要}
- 現状: {この repo の状態}
- 他 repo での実装: {どの repo にどうあるか}
- 汎用性: {この改善が tech stack 非依存か、固有の事情があるか}
- 取り込み提案: {具体的な変更内容}

### ギャップ 2: {概要}
...

## 対象: {owned_repo_2}
...

## note の不一致 (あれば)
- {repo}: note には「X」と書かれているが、実際には {Y} が存在する
```

### ステップ 5: 提案の確認と反映

各提案について AskUserQuestion で以下を選ばせる:

- **反映する (ローカルクローンあり)**: 対象リポジトリの `.claude/` を直接更新。変更をユーザーに表示し、コミットするかはユーザー判断
- **反映する (GitHub 上のみ)**: ローカルクローンがない場合、反映内容をブランチ + PR 用のコメントとして出力する (ユーザーが手動で適用)
- **スキップ**: 今回は反映しない
- **提案を却下**: 以後この提案を出さないよう、対象 repo の note に除外コメントを追記する (例: 「cross-review: X は独自実装のため取り込まない」)

### ステップ 6: note の更新 (任意)

ステップ 4 で note と現実の不一致を報告した repo について、ユーザーに note を更新するか確認する。

1. 不一致があった repo を列挙
2. 各 repo について「更新する / そのまま」を選ばせる
3. 「更新する」を選ばれた場合、更新案を提示して confirm 後に `${CLAUDE_PLUGIN_DATA}/registry.json` を更新する

## 注意

- owned=false の repo には一切書き込まない (PR を出せない前提)
- owned=true の repo への書き込みは必ずユーザー承認後に行う。勝手にコミット・プッシュしない
- この skill は registry の中身を双方向に使う。自分自身を除外する必要はない (cwd の概念がないため)
- tech stack 固有の記述 (ビルドコマンド等) は提案から除外する。汎用的な改善のみ対象
