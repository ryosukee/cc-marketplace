# impl-spec plugin skill 改善提案: async-progress テスト結果から

feedmarks の async-progress 案件で impl-spec plugin (requirements + design) を
テスト実行した結果のフィードバック
([レビュー結果](./impl-spec-feedback-async-progress.md)) を受け、
plugin skill のどこをどう修正すれば再発しないかを分析した。

## 問題のパターン分析

フィードバックの指摘を根本原因で分類すると 6 パターンに集約される。

| パターン | 根本原因 | 発生箇所 | 指摘件数 |
|---|---|---|---|
| 1. コードベース検索の不足 | Explore agent の出力を未検証で転記。「ない」の不在確認をしていない | requirements Phase 2, design Phase 2 | 6 件 (R1-R3, D2-D4) |
| 2. アーキテクチャルール未参照 | 擬似コードと `.claude/rules/` の整合チェック指示がない | design Phase 4 | 1 件 (D1) |
| 3. 既存コンポーネントの調査不足 | 新規ディレクトリ配置の規約整合、既存 store の移行パスが未調査 | design Phase 2 | 2 件 (D9-D10) |
| 4. 受入基準との整合チェック不足 | 「設計判断に委ねる」と書いた項目が受入基準と矛盾する可能性 | requirements Phase 4, design Phase 3 | 4 件 (R4-R7) |
| 5. 状態遷移の exhaustive check 不足 | state × event の全パターン列挙がない | design Phase 4 | 2 件 (D7-D8) |
| 6. 旧設計との退行比較なし | 旧設計で解決していた問題が新設計で退行 | design Phase 2 | 1 件 (D8 の一部) |

### パターン 1 の深掘り: なぜ agent 出力を検証しなかったか

requirements / design の skill 定義に「推測ではなく実装を読んで」と書いてあるが、
Explore agent に調査を委任した場合に「agent の出力を検証する」指示がない。

具体的に検出できなかったもの:

- SSE クライアント (`Sidebar.tsx`) — "EventSource" / "event-stream" / "ReadableStream" で grep すれば即発見
- `importJSON()` の不在 — `find_symbol importJSON` で即判明
- タグレコメンドの "Recommending..." テキスト — 対象 component の disabled 周辺を Read すれば発見
- Feed refresh ボタンの場所 — `Sidebar.tsx` を Read すれば判明

共通する構造: skill が「調査しろ」と言うだけで「結果を検証しろ」と言っていない。

## 修正提案

### requirements skill

#### Phase 2「調査の精度」セクション — 追記

以下を追記する:

```markdown
### 不在の検証

「存在しない」「〜していない」「〜のみ」と記述する前に、
grep / Serena MCP で不在を確認する。

例: 「frontend に SSE クライアントは存在しない」→
"EventSource" / "event-stream" / "ReadableStream" で grep して 0 件を確認。

Agent に委任した調査結果を転記する場合も、
否定的な記述 (「〜がない」「〜のみ」) は自分で検証してから書く。
Agent は「見つけられなかった」と「存在しない」を区別しないことがある。

### 現状記述の精度

スコープテーブル等で既存の振る舞いを記述する場合、
当該コンポーネントの該当行を Read して実装を確認する。

「button disabled のみ」と書くなら、disabled 以外の状態表示
(テキスト変化、spinner、badge 等) がないことまで確認する。

同一カテゴリとしてまとめる前に (例: 「Import は同期」)、
各 handler の実装パターンを個別に確認する。
同じ名前でも handler 内の分岐 (Content-Type 分岐、context.WithoutCancel の有無等) で
実質的に異なるパターンになっていることがある。
```

#### Phase 4「要件定義書の品質基準」セクション — 追記

以下を追記する:

```markdown
### 受入基準との整合検証

ファイルに書き出す前に、以下を検証する:

- 各受入基準に対して、機能要件の中に対応する具体的な定義があるか
- 「設計判断に委ねる」と書いた項目が、受入基準の検証条件と矛盾しないか
  例: 「保持期間は設計判断」と書きつつ受入基準に「リロード後も表示される」がある場合、
  「最低 N 分保持」等の下限を要件で決めないと受入基準が検証不能になる
- 矛盾がある場合は、機能要件を具体化するか受入基準を修正するかをユーザーに確認する

### テスト影響の概要

要件定義書のスコープ範囲に、以下の影響概要を記載する:

- スコープ内の変更が影響する既存テストの種類
  (例: handler テスト、mutation hook テスト、E2E の該当フロー)
- 廃止する endpoint / 型 / store に依存する既存テストの有無

テストの詳細計画は設計書の責務だが、
「影響が出る」という事実は要件レベルで把握しておく。
```

### design skill

#### Phase 2「調査対象」セクション — 追記

以下を追記する:

```markdown
### Agent 調査結果の検証

Agent (Explore) に調査を委任した場合、結果の中で以下に該当する記述は
自分で grep / Read / Serena MCP で検証する:

- 否定的な記述 (「〜は存在しない」「〜はない」「〜のみ」)
- ファイル名・関数名・handler 名の固有名詞
- ファイルパスと行番号 (特に削除・修正対象)

Agent は「見つけられなかった」と「存在しない」を区別しないことがある。
固有名詞は推測で生成されることがある。

### 新規ファイル・ディレクトリの配置規約整合

新規ファイルやディレクトリを追加する場合、
プロジェクトのディレクトリ構成規約 (CLAUDE.md, .claude/rules/ の
アーキテクチャルール等) と整合するか確認する。

規約に存在しないサブディレクトリを作る場合は、
設計判断として理由を記録するか、ユーザーに確認する。

### 旧版設計との退行比較

対象領域に既存の設計書・計画書・実装がある場合、それを Read して方針を把握する。
新設計で旧設計より退行するポイント
(堅牢性が下がる、カバー範囲が狭まる、既存で解決済みの問題が再発する等) がないか確認し、
退行がある場合は設計判断として理由を記録するか、ユーザーに確認する。
```

#### Phase 3「終了条件」セクション — 追記

以下を追記する:

```markdown
- 受入基準の各項目に対して、設計方針が矛盾なく対応しているか検証されている
```

#### Phase 4「設計書の品質基準」セクション — 追記

以下を追記する:

```markdown
### アーキテクチャルールとの整合チェック

設計書に擬似コード、コンポーネント配置、レイヤー間の呼び出し関係を書いた場合、
プロジェクトの `.claude/rules/` のアーキテクチャルールと矛盾しないか確認する。

例: handler 内での goroutine 起動、
service 層を介さない infra 直接参照、
アーキテクチャルールで規定された依存方向への違反など。

既存パターンから逸脱する場合は、設計判断の記録で理由を明記する。

### 状態遷移の exhaustive check

状態遷移を定義する場合、以下を検証する:

- 全状態 × 全イベントの組み合わせを列挙し、
  各組み合わせの振る舞い (遷移先 / エラー / no-op) を明記する
- 「未定義」の組み合わせを残さない
- 初期状態と終了状態を明示する
- カウンタ値のエッジケース (0/0、overflow 等) を確認する

### インターフェースのシグネチャ完全性

新規の interface / hook / API を定義する場合、以下を明記する:

- 全メソッド / 関数の完全なシグネチャ (引数名、引数型、戻り値型)
- 副作用の有無と内容 (例: Success() が内部カウンタをインクリメントするか)
- nil / undefined / ゼロ値の扱い
```

### spec-reviewer agent

#### チェック観点の追加

以下のチェック観点を spec-reviewer に追加する:

```markdown
### 不在記述の裏付け (doc_type=requirements, design 共通)

「〜は存在しない」「〜していない」「〜のみ」等の否定的な記述に対して、
それを裏付ける調査 (grep 結果、Serena MCP 結果) が言及されているか確認する。
裏付けなしの否定的記述は指摘する。

### 受入基準との整合 (doc_type=requirements)

受入基準の各項目に対して:

- 機能要件の中に対応する具体的な定義があるか
- 「設計判断に委ねる」とされた項目が受入基準と矛盾しないか
- 循環定義 (「A は B が必要な時間」「B は A が可能な期間」) がないか

### アーキテクチャルール違反 (doc_type=design)

擬似コード、コンポーネント配置、レイヤー間の呼び出し関係が
プロジェクトの `.claude/rules/` に違反していないか確認する。
プロジェクトのルールを読み取れない場合は「確認不可」として報告する。

### 状態遷移の完全性 (doc_type=design)

状態遷移定義がある場合、全状態 × 全イベントの組み合わせが
網羅されているか確認する。未定義の組み合わせがあれば指摘する。

### 新規ディレクトリの配置根拠 (doc_type=design)

新規ディレクトリを追加する場合、プロジェクトの既存構成規約
(CLAUDE.md / .claude/rules/) との整合が記述されているか確認する。
```

## 優先度

| 優先度 | 修正内容 | 対象 | 理由 |
|---|---|---|---|
| 高 | Agent 調査結果の検証 + 不在の検証 | requirements Phase 2, design Phase 2 | 指摘 6 件 (最多) の直接原因 |
| 高 | 受入基準との整合検証 | requirements Phase 4 | 指摘 4 件の直接原因 |
| 中 | アーキテクチャルール照合 | design Phase 4 | 1 件だが致命的 (ルール違反) |
| 中 | 状態遷移 exhaustive check | design Phase 4 | 2 件、エッジケース見落としの再発防止 |
| 中 | 新規ディレクトリ配置規約整合 | design Phase 2 | 2 件 |
| 低 | 旧設計との退行比較 | design Phase 2 | 1 件、頻度は低い |
| 低 | テスト影響の概要 | requirements Phase 4 | 旧版から未解決、要件レベルでは概要のみ |
